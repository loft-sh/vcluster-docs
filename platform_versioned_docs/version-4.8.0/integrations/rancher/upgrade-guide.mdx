:::info Legacy integration removed

The built-in Rancher integration was removed in vCluster Platform version v4.9.0. This page explains what happens when you upgrade an existing deployment and how to recover Tenant Clusters that become stuck. For the current Rancher integration, see [Install the Rancher integration for vCluster](/docs/vcluster/third-party-integrations/rancher/install-rancher-integration).

:::

## What changes on upgrade

When you upgrade a vCluster Platform instance that has the legacy Rancher integration configured to vCluster Platform v4.9.0 or later, the following happens automatically:

- **Running Tenant Clusters are unaffected.** Existing virtual clusters continue running normally.
- **Rancher sync no longer runs.** Member synchronization between Rancher and vCluster Platform is disabled.
- **Rancher auth is removed.** Users can no longer log in to vCluster Platform via Rancher SSO.
- **Rancher clusters stay active.** The Rancher cluster agent running inside each Tenant Cluster maintains its own direct connection to Rancher, independent of the Platform. Rancher clusters remain in `active` state after the upgrade.

:::note Want to upgrade your Tenant Clusters to use the new Rancher integration?

If you want to use the new operator-based Rancher integration, which will enable you to create new Tenant Clusters integrated with Rancher, see the [migration guide](./migration-guide.mdx).

:::

## Stuck Tenant Clusters after deletion

Tenant Clusters that had the Rancher integration enabled carry a `loft.sh/cleanup-rancher` finalizer on their `VirtualClusterInstance` object. After upgrading to v4.9.0 or later this finalizer is never processed — the controller that removes it no longer exists. This means **any deletion attempt on a Rancher-integrated Tenant Cluster will block indefinitely**.

You will see the Tenant Cluster in a `Terminating` state in the Platform UI and the Rancher cluster entry will show that it is in a disconnected or unavailable state.

### Identify affected Tenant Clusters

```bash
kubectl get virtualclusterinstances -A -o json | jq '
  .items[]
  | select(.metadata.finalizers[]? == "loft.sh/cleanup-rancher")
  | {name: .metadata.name, namespace: .metadata.namespace, phase: .status.phase}
'
```

### Unblock a stuck Tenant Cluster

Remove the finalizer manually to allow the deletion to complete:

<InterpolatedCodeBlock
  language="bash"
  code={"kubectl patch virtualclusterinstance [[VAR:NAME:my-tenant-cluster]] -n [[VAR:NAMESPACE:p-my-project]] \\\n  --type=json \\\n  -p='[{\"op\":\"remove\",\"path\":\"/metadata/finalizers/0\"}]'"}
/>

The Tenant Cluster will complete deletion once the finalizer is removed.

After the Tenant Cluster is deleted, the corresponding cluster entry in Rancher is not automatically removed — the Platform no longer communicates with Rancher. Verify the cluster shows as **Unavailable** or **Disconnected** in Rancher before deleting it. Go to **Cluster Management** in the Rancher UI, find the corresponding cluster entry, and delete it. If the deletion hangs, Rancher will eventually force-remove the entry once it detects the agent has been disconnected for a sufficient period of time.
