:::note
These instructions are tested on AWS. Multi-region platform can run on other cloud providers with comparable capabilities (managed Kubernetes, cross-region networking, latency-based DNS, and a shared database).
:::

These steps walk you through setting up vCluster Platform across two AWS regions using EKS, cross-region VPC
peering, shared database (Kine), ALB ingress, and Route 53 latency-based routing.

## Prerequisites

Before you begin, ensure you have:

-   An **AWS account** with sufficient IAM permissions
-   A **registered domain** (example: `multi-region.example.com`)
-   A **public Route 53 hosted zone**
-   The following tools installed:
    -   [`eksctl`](https://eksctl.io/installation/)
    -   [`kubectl`](https://kubernetes.io/docs/tasks/tools/)
    -   [`awscli`](https://docs.aws.amazon.com/cli/latest/userguide/getting-started-install.html)
    -   [`helm`](https://helm.sh/docs/intro/install/)
    -   [`vcluster`](/docs/vcluster/quick-start/shared-nodes#install-the-vcluster-cli)

## Step 1 - Create two EKS clusters

:::warning Important
All three VPCs (two EKS clusters and one database) must use **different, non-overlapping CIDR ranges** to allow VPC peering.
:::

**U.S. region**

<InterpolatedCodeBlock
  code={
`eksctl create cluster \\
  --name [[GLOBAL:US_CLUSTER_NAME]] \\
  --region [[GLOBAL:AWS_REGION]] \\
  --nodes 2 \\
  --managed \\
  --vpc-cidr [[GLOBAL:US_VPC_CIDR]] \\
  --with-oidc`
  }
  language="bash"
/>

**E.U. region**

<InterpolatedCodeBlock
  code={
`eksctl create cluster \\
  --name [[GLOBAL:EU_CLUSTER_NAME]] \\
  --region [[GLOBAL:EU_REGION]] \\
  --nodes 2 \\
  --managed \\
  --vpc-cidr [[GLOBAL:EU_VPC_CIDR]] \\
  --with-oidc`
  }
  language="bash"
/>

#### Install the Amazon EBS CSI driver

The EBS CSI driver is required for dynamic provisioning of persistent volumes
on EKS. Without it, tenant cluster StatefulSet pods remain in `Pending` state
because the `gp2` storage class can't provision volumes.

Install the driver as an EKS managed add-on on **each** cluster:

<InterpolatedCodeBlock
  code={
`eksctl create addon --name aws-ebs-csi-driver \\
  --cluster [[GLOBAL:US_CLUSTER_NAME]] \\
  --region [[GLOBAL:AWS_REGION]]

eksctl create addon --name aws-ebs-csi-driver \\
  --cluster [[GLOBAL:EU_CLUSTER_NAME]] \\
  --region [[GLOBAL:EU_REGION]]`
  }
  language="bash"
/>

For more details on EKS prerequisites for running tenant clusters, see the
[EKS environment setup guide](/docs/vcluster/next/deploy/control-plane/kubernetes-pod/environment/eks).

## Step 3 - Create the database (Kine backend)

Create an RDS instance (MariaDB) in its own **isolated VPC**, separate from the
EKS cluster VPCs. This keeps the database network boundary independent from the
cluster workloads and simplifies security group rules.

:::warning Important
For production multi-region deployments, configure RDS in an active-active setup so the database automatically fails over between regions. See [Active-Active Replication on Amazon RDS for MySQL](https://aws.amazon.com/solutions/guidance/active-active-replication-on-amazon-rds-for-mysql/) for guidance on setting this up.
:::

#### Create the database VPC

Create a VPC with a CIDR that doesn't overlap with either EKS cluster VPC.
Enable DNS support and DNS hostnames so the RDS endpoint resolves correctly
across VPC peering connections.

<InterpolatedCodeBlock
  code={
`aws ec2 create-vpc \\
  --cidr-block [[GLOBAL:DB_VPC_CIDR]] \\
  --region [[GLOBAL:AWS_REGION]] \\
  --tag-specifications 'ResourceType=vpc,Tags=[{Key=Name,Value=multi-region-db-vpc}]'`
  }
  language="bash"
/>

Note the `VpcId` from the output, then enable DNS settings:

<InterpolatedCodeBlock
  code={
`aws ec2 modify-vpc-attribute \\
  --vpc-id [[GLOBAL:DB_VPC_ID]] \\
  --enable-dns-support '{"Value":true}' \\
  --region [[GLOBAL:AWS_REGION]]

aws ec2 modify-vpc-attribute \\
  --vpc-id [[GLOBAL:DB_VPC_ID]] \\
  --enable-dns-hostnames '{"Value":true}' \\
  --region [[GLOBAL:AWS_REGION]]`
  }
  language="bash"
/>

#### Create subnets and DB subnet group

Create subnets in at least two availability zones (required by RDS), then create
a DB subnet group.

<InterpolatedCodeBlock
  code={
`aws ec2 create-subnet \\
  --vpc-id [[GLOBAL:DB_VPC_ID]] \\
  --cidr-block 10.1.1.0/24 \\
  --availability-zone us-east-1a \\
  --region [[GLOBAL:AWS_REGION]] \\
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=multi-region-db-subnet-1a}]'

aws ec2 create-subnet \\
  --vpc-id [[GLOBAL:DB_VPC_ID]] \\
  --cidr-block 10.1.2.0/24 \\
  --availability-zone us-east-1b \\
  --region [[GLOBAL:AWS_REGION]] \\
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=multi-region-db-subnet-1b}]'`
  }
  language="bash"
/>

<InterpolatedCodeBlock
  code={
`aws rds create-db-subnet-group \\
  --db-subnet-group-name multi-region-db-subnet \\
  --db-subnet-group-description "Isolated VPC subnet group for multi-region RDS" \\
  --subnet-ids [[VAR:DB_SUBNET_1A_ID:subnet-xxxxxxxxx]] [[VAR:DB_SUBNET_1B_ID:subnet-yyyyyyyyy]] \\
  --region [[GLOBAL:AWS_REGION]]`
  }
  language="bash"
/>

#### Create a security group for the database

Create a security group that allows inbound MariaDB traffic (port 3306) from
both EKS cluster VPC CIDR ranges.

<InterpolatedCodeBlock
  code={
`aws ec2 create-security-group \\
  --group-name multi-region-db-sg \\
  --description "Allow MariaDB from EKS VPCs via peering" \\
  --vpc-id [[GLOBAL:DB_VPC_ID]] \\
  --region [[GLOBAL:AWS_REGION]]`
  }
  language="bash"
/>

<InterpolatedCodeBlock
  code={
`aws ec2 authorize-security-group-ingress \\
  --group-id [[GLOBAL:DB_SG_ID]] \\
  --protocol tcp --port 3306 \\
  --cidr [[GLOBAL:US_VPC_CIDR]] \\
  --region [[GLOBAL:AWS_REGION]]

aws ec2 authorize-security-group-ingress \\
  --group-id [[GLOBAL:DB_SG_ID]] \\
  --protocol tcp --port 3306 \\
  --cidr [[GLOBAL:EU_VPC_CIDR]] \\
  --region [[GLOBAL:AWS_REGION]]`
  }
  language="bash"
/>

#### Create the RDS instance

<InterpolatedCodeBlock
  code={
`aws rds create-db-instance \\
  --engine mariadb \\
  --db-instance-identifier mariadb-multi-region \\
  --allocated-storage 20 \\
  --region [[GLOBAL:AWS_REGION]] \\
  --db-instance-class db.t3.medium \\
  --master-username admin \\
  --master-user-password [[VAR:ADMIN_PASSWORD:your-password]] \\
  --db-subnet-group-name multi-region-db-subnet \\
  --vpc-security-group-ids [[GLOBAL:DB_SG_ID]] \\
  --no-publicly-accessible \\
  --enable-iam-database-authentication`
  }
  language="bash"
/>

Wait for the instance to become available:

<InterpolatedCodeBlock
  code={
`aws rds wait db-instance-available \\
  --db-instance-identifier mariadb-multi-region \\
  --region [[GLOBAL:AWS_REGION]]`
  }
  language="bash"
/>

#### Create the Kine database and IAM user

Connect to the database instance as the `admin` user and create the `kine`
database and user. The user authenticates with the `AWSAuthenticationPlugin`
instead of a password.

```sql
CREATE DATABASE IF NOT EXISTS kine;

CREATE USER 'kine'@'%' IDENTIFIED WITH AWSAuthenticationPlugin AS 'RDS';
GRANT ALL PRIVILEGES ON kine.* TO 'kine'@'%';
FLUSH PRIVILEGES;
```

#### Create an IAM policy for RDS access

Create an IAM policy that grants the `rds-db:connect` permission for the `kine`
database user. Replace `<DBI_RESOURCE_ID>` with the RDS instance resource ID (run
`aws rds describe-db-instances --query 'DBInstances[0].DbiResourceId'` to retrieve it).

<InterpolatedCodeBlock
  code={
`aws iam create-policy \\
  --policy-name RDSIAMAuthKine \\
  --policy-document '{
    "Version": "2012-10-17",
    "Statement": [
      {
        "Effect": "Allow",
        "Action": "rds-db:connect",
        "Resource": "arn:aws:rds-db:[[GLOBAL:AWS_REGION]]:[[GLOBAL:ACCOUNT_ID]]:dbuser:[[VAR:DBI_RESOURCE_ID:db-XXXXXXXXXXXXXXXXXXXXXXXXXX]]/kine"
      }
    ]
  }'`
  }
  language="bash"
/>

#### Create an IAM role for Pod Identity

Create an IAM role with a trust policy that allows the EKS Pod Identity agent
to assume it. The `pods.eks.amazonaws.com` trust principal is region-agnostic,
so a single role works for clusters in any region.

<InterpolatedCodeBlock
  code={
`aws iam create-role \\
  --role-name PlatformKineRDSRole \\
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [
      {
        "Effect": "Allow",
        "Principal": {
          "Service": "pods.eks.amazonaws.com"
        },
        "Action": [
          "sts:AssumeRole",
          "sts:TagSession"
        ]
      }
    ]
  }'

aws iam attach-role-policy \\
  --role-name PlatformKineRDSRole \\
  --policy-arn arn:aws:iam::[[GLOBAL:ACCOUNT_ID]]:policy/RDSIAMAuthKine`
  }
  language="bash"
/>

#### Install the EKS Pod Identity Agent

Install the Pod Identity Agent add-on on **every** cluster. Without this
add-on, the Pod Identity association has no effect and pods can't obtain
IAM credentials.

<InterpolatedCodeBlock
  code={
`aws eks create-addon \\
  --cluster-name [[GLOBAL:US_CLUSTER_NAME]] \\
  --addon-name eks-pod-identity-agent \\
  --region [[GLOBAL:AWS_REGION]]

aws eks create-addon \\
  --cluster-name [[GLOBAL:EU_CLUSTER_NAME]] \\
  --addon-name eks-pod-identity-agent \\
  --region [[GLOBAL:EU_REGION]]`
  }
  language="bash"
/>

Verify the add-on is `ACTIVE` on each cluster before proceeding:

<InterpolatedCodeBlock
  code={
`aws eks describe-addon \\
  --cluster-name [[GLOBAL:US_CLUSTER_NAME]] \\
  --addon-name eks-pod-identity-agent \\
  --region [[GLOBAL:AWS_REGION]] \\
  --query 'addon.status'`
  }
  language="bash"
/>

#### Create Pod Identity associations

Associate the IAM role with the `loft` service account in the
`vcluster-platform` namespace on **every** cluster. This allows the platform
pods to generate temporary IAM authentication tokens for the database
connection.

:::warning Important
You must create a Pod Identity association on **each** cluster, not just the
cluster in the same region as the database. Without the association, the
platform pods in that cluster can't authenticate with the RDS instance and
fail with `Access denied` errors.
:::

**U.S. region**

<InterpolatedCodeBlock
  code={
`aws eks create-pod-identity-association \\
  --cluster-name [[GLOBAL:US_CLUSTER_NAME]] \\
  --role-arn arn:aws:iam::[[GLOBAL:ACCOUNT_ID]]:role/PlatformKineRDSRole \\
  --namespace vcluster-platform \\
  --service-account loft \\
  --region [[GLOBAL:AWS_REGION]]`
  }
  language="bash"
/>

**E.U. region**

<InterpolatedCodeBlock
  code={
`aws eks create-pod-identity-association \\
  --cluster-name [[GLOBAL:EU_CLUSTER_NAME]] \\
  --role-arn arn:aws:iam::[[GLOBAL:ACCOUNT_ID]]:role/PlatformKineRDSRole \\
  --namespace vcluster-platform \\
  --service-account loft \\
  --region [[GLOBAL:EU_REGION]]`
  }
  language="bash"
/>

For more details on IAM database authentication with EKS Pod Identity, see
[IAM database authentication](/docs/platform/administer/connector/iam_database).

## Step 4 - Create VPC peering to the database

Create VPC peering connections between each EKS cluster VPC and the database
VPC. This gives both clusters network access to the RDS instance without
requiring direct cluster-to-cluster peering.

#### Create peering connections

**Database VPC to U.S. cluster VPC** (same region):

<InterpolatedCodeBlock
  code={
`aws ec2 create-vpc-peering-connection \\
  --vpc-id [[GLOBAL:DB_VPC_ID]] \\
  --peer-vpc-id [[VAR:VPC_ID_US_CLUSTER:vpc-yyyyyyyyy]] \\
  --region [[GLOBAL:AWS_REGION]] \\
  --tag-specifications 'ResourceType=vpc-peering-connection,Tags=[{Key=Name,Value=db-vpc-to-eks-[[GLOBAL:AWS_REGION]]}]'`
  }
  language="bash"
/>

**Database VPC to E.U. cluster VPC** (cross-region):

<InterpolatedCodeBlock
  code={
`aws ec2 create-vpc-peering-connection \\
  --vpc-id [[GLOBAL:DB_VPC_ID]] \\
  --peer-vpc-id [[VAR:VPC_ID_EU_CLUSTER:vpc-zzzzzzzzz]] \\
  --peer-region [[GLOBAL:EU_REGION]] \\
  --region [[GLOBAL:AWS_REGION]] \\
  --tag-specifications 'ResourceType=vpc-peering-connection,Tags=[{Key=Name,Value=db-vpc-to-eks-[[GLOBAL:EU_REGION]]}]'`
  }
  language="bash"
/>

#### Accept the peering connections

The same-region peering is accepted in `us-east-1`. The cross-region peering
must be accepted in `eu-west-1`.

<InterpolatedCodeBlock
  code={
`aws ec2 accept-vpc-peering-connection \\
  --vpc-peering-connection-id [[GLOBAL:US_PEERING_ID]] \\
  --region [[GLOBAL:AWS_REGION]]

aws ec2 accept-vpc-peering-connection \\
  --vpc-peering-connection-id [[GLOBAL:EU_PEERING_ID]] \\
  --region [[GLOBAL:EU_REGION]]`
  }
  language="bash"
/>

#### Enable DNS resolution

Enable DNS resolution on both peering connections so the RDS endpoint hostname
resolves to its private IP from the EKS VPCs.

<InterpolatedCodeBlock
  code={
`aws ec2 modify-vpc-peering-connection-options \\
  --vpc-peering-connection-id [[GLOBAL:US_PEERING_ID]] \\
  --requester-peering-connection-options AllowDnsResolutionFromRemoteVpc=true \\
  --accepter-peering-connection-options AllowDnsResolutionFromRemoteVpc=true \\
  --region [[GLOBAL:AWS_REGION]]`
  }
  language="bash"
/>

For the cross-region peering, set the requester side (DB VPC) in `us-east-1`
and the accepter side (E.U. cluster VPC) in `eu-west-1`:

<InterpolatedCodeBlock
  code={
`aws ec2 modify-vpc-peering-connection-options \\
  --vpc-peering-connection-id [[GLOBAL:EU_PEERING_ID]] \\
  --requester-peering-connection-options AllowDnsResolutionFromRemoteVpc=true \\
  --region [[GLOBAL:AWS_REGION]]

aws ec2 modify-vpc-peering-connection-options \\
  --vpc-peering-connection-id [[GLOBAL:EU_PEERING_ID]] \\
  --accepter-peering-connection-options AllowDnsResolutionFromRemoteVpc=true \\
  --region [[GLOBAL:EU_REGION]]`
  }
  language="bash"
/>

#### Add routes

Add routes so traffic can flow between each EKS VPC and the database VPC.

:::warning Important
Add routes to **every** route table whose subnets host EKS nodes, including
public route tables. If your EKS nodes run in public subnets (the `eksctl`
default), missing routes on the public route table cause database connection
timeouts even though VPC peering is active.
:::

**Database VPC route table** — route to both EKS VPC CIDRs:

<InterpolatedCodeBlock
  code={
`aws ec2 create-route \\
  --route-table-id [[GLOBAL:DB_VPC_ROUTE_TABLE_ID]] \\
  --destination-cidr-block [[GLOBAL:US_VPC_CIDR]] \\
  --vpc-peering-connection-id [[GLOBAL:US_PEERING_ID]] \\
  --region [[GLOBAL:AWS_REGION]]

aws ec2 create-route \\
  --route-table-id [[GLOBAL:DB_VPC_ROUTE_TABLE_ID]] \\
  --destination-cidr-block [[GLOBAL:EU_VPC_CIDR]] \\
  --vpc-peering-connection-id [[GLOBAL:EU_PEERING_ID]] \\
  --region [[GLOBAL:AWS_REGION]]`
  }
  language="bash"
/>

**U.S. EKS VPC route tables** — route to the database VPC CIDR. Repeat for
every route table associated with subnets that host EKS nodes:

<InterpolatedCodeBlock
  code={
`aws ec2 create-route \\
  --route-table-id [[VAR:US_EKS_ROUTE_TABLE_ID:rtb-yyyyyyyyy]] \\
  --destination-cidr-block [[GLOBAL:DB_VPC_CIDR]] \\
  --vpc-peering-connection-id [[GLOBAL:US_PEERING_ID]] \\
  --region [[GLOBAL:AWS_REGION]]`
  }
  language="bash"
/>

**E.U. EKS VPC route tables** — route to the database VPC CIDR. Repeat for
every route table associated with subnets that host EKS nodes:

<InterpolatedCodeBlock
  code={
`aws ec2 create-route \\
  --route-table-id [[VAR:EU_EKS_ROUTE_TABLE_ID:rtb-zzzzzzzzz]] \\
  --destination-cidr-block [[GLOBAL:DB_VPC_CIDR]] \\
  --vpc-peering-connection-id [[GLOBAL:EU_PEERING_ID]] \\
  --region [[GLOBAL:EU_REGION]]`
  }
  language="bash"
/>

To list all route tables for a VPC and identify which subnets they serve:

<InterpolatedCodeBlock
  code={
`aws ec2 describe-route-tables \\
  --filters "Name=vpc-id,Values=[[VAR:VPC_ID:vpc-xxxxxxxxx]]" \\
  --region [[GLOBAL:AWS_REGION]] \\
  --query 'RouteTables[].{RouteTableId:RouteTableId,Name:Tags[?Key==\`Name\`].Value|[0],Subnets:Associations[].SubnetId}'`
  }
  language="bash"
/>

#### Verify connectivity

Launch a test pod in each EKS cluster and verify it can reach the RDS endpoint:

<InterpolatedCodeBlock
  code={
`kubectl run dbtest --image=busybox --restart=Never --rm -it -- \\
  nc -zv [[VAR:RDS_ENDPOINT:mariadb-multi-region.xxxxxxxxxxxx.us-east-1.rds.amazonaws.com]] 3306`
  }
  language="bash"
/>

The connection should report `open`. If it times out, verify that routes exist
on the correct route tables and that the database security group allows ingress
from the cluster's VPC CIDR.

## Step 5 - Deploy vCluster platform

Deploy the platform on each cluster using the `multiRegion` Helm values. Each cluster needs a **region-specific** `multiRegion.host` value (used as the custom DERP server endpoint), while `config.loftHost` should be the **shared DNS domain**.

Create a values file for each region. The only differences between regions are
`multiRegion.host` and `multiRegion.region`.

**U.S. region** (save as `US_VALUES_FILE`)

<InterpolatedCodeBlock
  code={"admin:\n  email: [[GLOBAL:ADMIN_EMAIL]]\n\nmultiRegion:\n  enabled: true\n  host: us.multi-region.example.com\n  region: [[GLOBAL:AWS_REGION]]\n\nreplicaCount: 3\n\nconfig:\n  loftHost: multi-region.example.com\n  database:\n    enabled: true\n    dataSource: \"mysql://kine@tcp([[GLOBAL:DATABASE_URL]]:3306)/kine\"\n    identityProvider: \"aws\"\n    extraArgs:\n      - --datastore-max-open-connections=20\n      - --datastore-max-idle-connections=0\n\n  # Cost control requires a single-region database and is not compatible\n  # with the shared Kine backend used in multi-region deployments.\n  costControl:\n    enabled: false\n\nagentValues:\n  replicaCount: 3"}
  language="yaml"
/>

## Step 6 - Configure HTTPS with AWS Certificate Manager (ACM)

Request a certificate in each region that covers both the shared DNS domain
and the region-specific subdomain. The wildcard `*.multi-region.example.com`
covers the regional subdomains (`us.multi-region.example.com`,
`eu.multi-region.example.com`), and `multi-region.example.com` is added as a
SAN to cover the apex.

Run this command in each region. The command uses the `AWS_REGION` page variable — run it a second time after setting `EU_REGION` as the region value to request the certificate there too.

<InterpolatedCodeBlock
  code={
`aws acm request-certificate \\
  --region [[GLOBAL:AWS_REGION]] \\
  --domain-name "multi-region.example.com" \\
  --subject-alternative-names "*.multi-region.example.com" \\
  --validation-method DNS`
  }
  language="bash"
/>

Get the certificate ARN from the output and describe the certificate.
Replace `<CERTIFICATE_ARN_ID>` with the ARN ID generated from the command above.

<InterpolatedCodeBlock
  code={
`aws acm describe-certificate \\
  --region [[GLOBAL:AWS_REGION]] \\
  --certificate-arn arn:aws:acm:[[GLOBAL:AWS_REGION]]:[[GLOBAL:ACCOUNT_ID]]:certificate/[[VAR:CERTIFICATE_ARN_ID:xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx]]`
  }
  language="bash"
/>

Add DNS validation CNAMEs in Route 53 and wait until certificate status is ISSUED.

:::note
You can also do this step by setting up cert-manager in the clusters instead of using AWS ACM.
:::

## Step 8 - Configure Route 53 latency-based routing

#### Create health checks

Create an HTTPS health check for each regional ALB. Route 53 uses these to
detect when a region is unavailable and fail over traffic automatically.

**U.S. region**

<InterpolatedCodeBlock
  code={
`aws route53 create-health-check \\
  --caller-reference "[[GLOBAL:AWS_REGION]]-platform-healthz" \\
  --health-check-config '{
    "FullyQualifiedDomainName": "us.[[GLOBAL:PLATFORM_URL]]",
    "Port": 443,
    "Type": "HTTPS",
    "ResourcePath": "/healthz",
    "RequestInterval": 10,
    "FailureThreshold": 3,
    "EnableSNI": true
  }'`
  }
  language="bash"
/>

**E.U. region**

<InterpolatedCodeBlock
  code={
`aws route53 create-health-check \\
  --caller-reference "[[GLOBAL:EU_REGION]]-platform-healthz" \\
  --health-check-config '{
    "FullyQualifiedDomainName": "eu.[[GLOBAL:PLATFORM_URL]]",
    "Port": 443,
    "Type": "HTTPS",
    "ResourcePath": "/healthz",
    "RequestInterval": 10,
    "FailureThreshold": 3,
    "EnableSNI": true
  }'`
  }
  language="bash"
/>

Note the health check IDs from the output. You can tag them for easier identification:

<InterpolatedCodeBlock
  code={
`aws route53 change-tags-for-resource \\
  --resource-type healthcheck --resource-id [[GLOBAL:HEALTH_CHECK_ID]] \\
  --add-tags Key=Name,Value=[[GLOBAL:REGION]]-platform-healthz`
  }
  language="bash"
/>

#### Create DNS records

Create latency-based A (Alias) records in Route 53 pointing to each
regional ALB. Attach the health checks created above so that Route 53 can
fail over traffic when a region becomes unavailable.

Replace `<US_HEALTH_CHECK_ID>` and `<EU_HEALTH_CHECK_ID>` with the IDs from the
previous step, and `<US_ALB_HOSTED_ZONE_ID>`, `<EU_ALB_HOSTED_ZONE_ID>` with the
canonical hosted zone IDs of each ALB.

<InterpolatedCodeBlock
  code={
`aws route53 change-resource-record-sets \\
  --hosted-zone-id [[GLOBAL:HOSTED_ZONE_ID]] \\
  --change-batch '{
    "Changes": [
      {
        "Action": "UPSERT",
        "ResourceRecordSet": {
          "Name": "multi-region.example.com",
          "Type": "A",
          "SetIdentifier": "[[GLOBAL:AWS_REGION]]",
          "Region": "[[GLOBAL:AWS_REGION]]",
          "HealthCheckId": "[[VAR:US_HEALTH_CHECK_ID:xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx]]",
          "AliasTarget": {
            "HostedZoneId": "[[GLOBAL:US_ALB_HOSTED_ZONE_ID]]",
            "DNSName": "[[GLOBAL:US_ALB_DNS_NAME]]",
            "EvaluateTargetHealth": true
          }
        }
      },
      {
        "Action": "UPSERT",
        "ResourceRecordSet": {
          "Name": "multi-region.example.com",
          "Type": "A",
          "SetIdentifier": "[[GLOBAL:EU_REGION]]",
          "Region": "[[GLOBAL:EU_REGION]]",
          "HealthCheckId": "[[VAR:EU_HEALTH_CHECK_ID:yyyyyyyy-yyyy-yyyy-yyyy-yyyyyyyyyyyy]]",
          "AliasTarget": {
            "HostedZoneId": "[[GLOBAL:EU_ALB_HOSTED_ZONE_ID]]",
            "DNSName": "[[GLOBAL:EU_ALB_DNS_NAME]]",
            "EvaluateTargetHealth": true
          }
        }

Confirm both health checks are healthy before proceeding:

<InterpolatedCodeBlock
  code={

## Step 9 - Deploy the agent on the clusters

The agent must be deployed in a **different namespace** than the platform (for example, `vcluster-agent`). The `agentValues` block in the platform values files (Step 5) controls the agent's replica count, leader election, and other settings.

<Tabs
  defaultValue="cli"
  values={[
    { label: "vCluster CLI", value: "cli" },
    { label: "Helm", value: "helm" },
  ]}
>
  <TabItem value="cli">

Use the vCluster CLI to add each cluster as a connected cluster. Generate an access token from the platform UI under **Users & Teams > Access Keys**, or by running `vcluster platform get token`.

<InterpolatedCodeBlock
  code={
`vcluster platform add cluster [[GLOBAL:US_CLUSTER_NAME]] \\
  --namespace vcluster-agent \\
  --display-name [[VAR:DISPLAY_NAME:my-cluster]] \\
  --values 'https://[[GLOBAL:PLATFORM_URL]]/clusters/agent-values/[[GLOBAL:US_CLUSTER_NAME]]?token=[[GLOBAL:ACCESS_TOKEN]]'`
  }
  language="bash"
/>

Repeat on the second cluster, substituting `EU_CLUSTER_NAME` for the cluster name and providing a new `DISPLAY_NAME`.

  </TabItem>
  <TabItem value="helm">

Go to the platform UI and under Infra -> Clusters, add a connected cluster and select the Helm (advanced) option.

You get a Helm command with a `--values` URL that contains agent-specific configuration.

<InterpolatedCodeBlock

  --set agentOnly=true \\
  --set url=[[GLOBAL:PLATFORM_URL]] \\
  --set token=[[GLOBAL:ACCESS_TOKEN]] \\

  }
  language="bash"
/>

Repeat on the second cluster, substituting `EU_CLUSTER_NAME` for the cluster name.

  </TabItem>

### Result

You now have a fully operational **multi-region vCluster Platform
deployment** with:

-   Two EKS clusters (U.S. + E.U.)
-   Shared Kine database in an isolated VPC with IAM-based authentication (Pod Identity)
-   VPC peering between each cluster VPC and the database VPC
-   Regional ALBs with extended idle timeout
-   ACM wildcard certificates
-   Route 53 latency-based routing with health checks for automatic failover
-   Platform deployed on both clusters sharing the same Kine database
