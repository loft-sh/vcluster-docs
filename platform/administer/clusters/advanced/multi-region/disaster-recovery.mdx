## Restore the Kine database from a snapshot

This runbook describes how to restore the shared Kine database from an RDS
snapshot to a new instance and update both platform regions to use the restored
database. Use this procedure for disaster recovery, database migration (for
example, enabling IAM authentication), or point-in-time recovery.

:::warning Important
Both platform regions must be scaled down before switching the data source to
prevent split-brain writes to the old and new databases.
:::

### Step 1 - Create a snapshot of the current database

Skip this step if you already have a snapshot to restore from.

<InterpolatedCodeBlock
  code={
`aws rds create-db-snapshot \\
  --db-instance-identifier [[VAR:CURRENT_DB_INSTANCE_ID:mariadb-multi-region]] \\
  --db-snapshot-identifier [[VAR:SNAPSHOT_NAME:kine-backup-YYYY-MM-DD]] \\
  --region [[GLOBAL:AWS_REGION]]`
  }
  language="bash"
/>

Wait for the snapshot to become available:

<InterpolatedCodeBlock
  code={
`aws rds wait db-snapshot-available \\
  --db-snapshot-identifier [[VAR:SNAPSHOT_NAME:kine-backup-YYYY-MM-DD]] \\
  --region [[GLOBAL:AWS_REGION]]`
  }
  language="bash"
/>

### Step 2 - Restore the snapshot to a new RDS instance

Restore the snapshot to a new instance in the database VPC. Use the same DB
subnet group and security group from the original setup. Include
`--enable-iam-database-authentication` if the new instance should use IAM auth.

<InterpolatedCodeBlock
  code={
`aws rds restore-db-instance-from-db-snapshot \\
  --db-instance-identifier [[VAR:NEW_DB_INSTANCE_ID:mariadb-multi-region-restored]] \\
  --db-snapshot-identifier [[VAR:SNAPSHOT_NAME:kine-backup-YYYY-MM-DD]] \\
  --db-instance-class db.t3.medium \\
  --db-subnet-group-name multi-region-db-subnet \\
  --vpc-security-group-ids [[GLOBAL:DB_SG_ID]] \\
  --no-publicly-accessible \\
  --enable-iam-database-authentication \\
  --region [[GLOBAL:AWS_REGION]]`
  }
  language="bash"
/>

Wait for the new instance to become available:

<InterpolatedCodeBlock
  code={
`aws rds wait db-instance-available \\
  --db-instance-identifier [[VAR:NEW_DB_INSTANCE_ID:mariadb-multi-region-restored]] \\
  --region [[GLOBAL:AWS_REGION]]`
  }
  language="bash"
/>

Note the new endpoint:

<InterpolatedCodeBlock
  code={
`aws rds describe-db-instances \\
  --db-instance-identifier [[VAR:NEW_DB_INSTANCE_ID:mariadb-multi-region-restored]] \\
  --query 'DBInstances[0].Endpoint.Address' \\
  --output text \\
  --region [[GLOBAL:AWS_REGION]]`
  }
  language="bash"
/>

If using IAM authentication, note the `DbiResourceId` of the new instance and
update the `RDSIAMAuthKine` IAM policy to include it. Without this, platform
pods fail with `Access denied for user 'kine'` errors because the IAM
`rds-db:connect` permission is scoped to a specific RDS instance resource ID.

<InterpolatedCodeBlock
  code={
`aws rds describe-db-instances \\
  --db-instance-identifier [[VAR:NEW_DB_INSTANCE_ID:mariadb-multi-region-restored]] \\
  --query 'DBInstances[0].DbiResourceId' \\
  --output text \\
  --region [[GLOBAL:AWS_REGION]]`
  }
  language="bash"
/>

Add the new resource ID to the policy's `Resource` array:

<InterpolatedCodeBlock
  code={
`aws iam create-policy-version \\
  --policy-arn arn:aws:iam::[[GLOBAL:ACCOUNT_ID]]:policy/RDSIAMAuthKine \\
  --set-as-default \\
  --policy-document '{
    "Version": "2012-10-17",
    "Statement": [
      {
        "Effect": "Allow",
        "Action": "rds-db:connect",
        "Resource": [
          "arn:aws:rds-db:[[GLOBAL:AWS_REGION]]:[[GLOBAL:ACCOUNT_ID]]:dbuser:[[VAR:OLD_DBI_RESOURCE_ID:db-OLDXXXXXXXXXXXXXXXXXXXXXXXXXX]]/kine",
          "arn:aws:rds-db:[[GLOBAL:AWS_REGION]]:[[GLOBAL:ACCOUNT_ID]]:dbuser:[[VAR:NEW_DBI_RESOURCE_ID:db-NEWXXXXXXXXXXXXXXXXXXXXXXXXXX]]/kine"
        ]
      }
    ]
  }'`
  }
  language="bash"
/>

### Step 3 - Scale down both regions

Scale both platform deployments to zero to stop all writes to the old database.

<InterpolatedCodeBlock
  code={
`kubectl --context [[GLOBAL:FIRST_REGION_CONTEXT]] \\
  scale deployment -n vcluster-platform loft --replicas=0

kubectl --context [[GLOBAL:SECOND_REGION_CONTEXT]] \\
  scale deployment -n vcluster-platform loft --replicas=0`
  }
  language="bash"
/>

Wait for all pods to stop:

<InterpolatedCodeBlock
  code={
`kubectl --context [[GLOBAL:FIRST_REGION_CONTEXT]] \\
  get pods -n vcluster-platform -l app=loft --watch

kubectl --context [[GLOBAL:SECOND_REGION_CONTEXT]] \\
  get pods -n vcluster-platform -l app=loft --watch`
  }
  language="bash"
/>

### Step 4 - Update the values files

Update the `dataSource` in both region values files to point to the new RDS
endpoint:

<InterpolatedCodeBlock
  code={"config:\n  database:\n    dataSource: \"mysql://kine@tcp([[VAR:NEW_DATABASE_URL:mariadb-multi-region-restored.xxxxxxxxxxxx.us-east-1.rds.amazonaws.com]]:3306)/kine\""}
  language="yaml"
/>

### Step 5 - Upgrade both regions

Apply the updated values files to both regions.

<Tabs
  defaultValue="cli"
  values={[
    { label: "vCluster CLI", value: "cli" },
    { label: "Helm", value: "helm" },
  ]}
>
  <TabItem value="cli">

<InterpolatedCodeBlock
  code={
`vcluster platform start \\
  --namespace vcluster-platform \\
  --kube-context [[GLOBAL:FIRST_REGION_CONTEXT]] \\
  --values [[VAR:FIRST_REGION_VALUES_FILE:platform-us-east-1-values.yaml]] \\
  --upgrade \\
  --no-tunnel

vcluster platform start \\
  --namespace vcluster-platform \\
  --kube-context [[GLOBAL:SECOND_REGION_CONTEXT]] \\
  --values [[VAR:SECOND_REGION_VALUES_FILE:platform-eu-west-1-values.yaml]] \\
  --upgrade \\
  --no-tunnel`
  }
  language="bash"
/>

  </TabItem>
  <TabItem value="helm">

<InterpolatedCodeBlock
  code={
`helm upgrade loft vcluster-platform --install --create-namespace --repository-config='' \\
  --namespace vcluster-platform \\
  --repo "https://charts.loft.sh/" \\
  --version [[VAR:CHART_VERSION:4.8.0]] \\
  --kube-context [[GLOBAL:FIRST_REGION_CONTEXT]] \\
  -f [[VAR:FIRST_REGION_VALUES_FILE:platform-us-east-1-values.yaml]] \\
  --server-side=true --force-conflicts

helm upgrade loft vcluster-platform --install --create-namespace --repository-config='' \\
  --namespace vcluster-platform \\
  --repo "https://charts.loft.sh/" \\
  --version [[VAR:CHART_VERSION:4.8.0]] \\
  --kube-context [[GLOBAL:SECOND_REGION_CONTEXT]] \\
  -f [[VAR:SECOND_REGION_VALUES_FILE:platform-eu-west-1-values.yaml]] \\
  --server-side=true --force-conflicts`
  }
  language="bash"
/>

  </TabItem>
</Tabs>

### Step 6 - Scale up both regions

The Helm upgrade keeps replicas at zero because it does not override the manual
scale-down. Scale both regions back up.

<InterpolatedCodeBlock
  code={
`kubectl --context [[GLOBAL:FIRST_REGION_CONTEXT]] \\
  scale deployment -n vcluster-platform loft --replicas=[[VAR:REPLICA_COUNT:3]]

kubectl --context [[GLOBAL:SECOND_REGION_CONTEXT]] \\
  scale deployment -n vcluster-platform loft --replicas=[[VAR:REPLICA_COUNT:3]]`
  }
  language="bash"
/>

Wait for all pods to become ready:

<InterpolatedCodeBlock
  code={
`kubectl --context [[GLOBAL:FIRST_REGION_CONTEXT]] \\
  rollout status deployment/loft -n vcluster-platform

kubectl --context [[GLOBAL:SECOND_REGION_CONTEXT]] \\
  rollout status deployment/loft -n vcluster-platform`
  }
  language="bash"
/>

### Step 7 - Verify the restore

Confirm the platform is healthy on both regions:

<InterpolatedCodeBlock
  code={
`for CTX in [[GLOBAL:FIRST_REGION_CONTEXT]] [[GLOBAL:SECOND_REGION_CONTEXT]]; do
  echo "=== $CTX ==="
  kubectl --context "$CTX" get pods -n vcluster-platform -l app=loft
  echo
done`
  }
  language="bash"
/>

Verify the platform UI is accessible through the shared DNS domain and that both
Route 53 health checks return healthy.

## Recover from regional failover

When a region goes down, Route 53 health checks detect the failure and
automatically redirect traffic to the healthy region. This runbook describes how
to diagnose the failed region, restore it, and confirm that both regions are
serving traffic again.

### Step 1 - Confirm failover is active

Verify which health check is failing:

<InterpolatedCodeBlock
  code={
`aws route53 get-health-check-status --health-check-id [[VAR:FAILED_REGION_HEALTH_CHECK_ID:xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx]] \\
  --query 'HealthCheckObservations[].{Region:Region,Status:StatusReport.Status}' \\
  --output table`
  }
  language="bash"
/>

Confirm the healthy region is still passing:

<InterpolatedCodeBlock
  code={
`aws route53 get-health-check-status --health-check-id [[VAR:HEALTHY_REGION_HEALTH_CHECK_ID:yyyyyyyy-yyyy-yyyy-yyyy-yyyyyyyyyyyy]] \\
  --query 'HealthCheckObservations[].{Region:Region,Status:StatusReport.Status}' \\
  --output table`
  }
  language="bash"
/>

:::note
During failover, the platform continues to operate normally through the healthy
region. Users may experience slightly higher latency if they are geographically
closer to the failed region.
:::

### Step 2 - Diagnose the failed region

Check the state of the platform pods in the failed region:

<InterpolatedCodeBlock
  code={
`kubectl --context [[VAR:FAILED_REGION_CONTEXT:arn:aws:eks:us-east-1:123456789012:cluster/platform-multi-region-us-east-1]] \\
  get pods -n vcluster-platform -l app=loft`
  }
  language="bash"
/>

Check pod logs for errors:

<InterpolatedCodeBlock
  code={
`kubectl --context [[VAR:FAILED_REGION_CONTEXT:arn:aws:eks:us-east-1:123456789012:cluster/platform-multi-region-us-east-1]] \\
  logs -n vcluster-platform -l app=loft --tail=50`
  }
  language="bash"
/>

Check the ALB and ingress status:

<InterpolatedCodeBlock
  code={
`kubectl --context [[VAR:FAILED_REGION_CONTEXT:arn:aws:eks:us-east-1:123456789012:cluster/platform-multi-region-us-east-1]] \\
  get ingress -n vcluster-platform loft`
  }
  language="bash"
/>

Common failure causes:
- **Pod crashes (CrashLoopBackOff)**: Check logs for database connectivity
  errors or resource exhaustion.
- **Node failures**: Check `kubectl get nodes` for `NotReady` nodes.
- **ALB unhealthy**: Verify the ALB target group health in the AWS console or
  with `aws elbv2 describe-target-health`.
- **Network partition**: Verify that VPC peering between the cluster VPC and
  the database VPC is active, that routes exist on the correct route tables
  (including public route tables if nodes are in public subnets), and that the
  database security group allows port 3306 from the cluster VPC CIDR.

### Step 3 - Restore the failed region

The recovery steps depend on the failure cause.

**If pods are crashlooping or unhealthy**, restart the deployment:

<InterpolatedCodeBlock
  code={
`kubectl --context [[VAR:FAILED_REGION_CONTEXT:arn:aws:eks:us-east-1:123456789012:cluster/platform-multi-region-us-east-1]] \\
  rollout restart deployment/loft -n vcluster-platform`
  }
  language="bash"
/>

**If the deployment is scaled to zero or missing**, re-apply the values file:

<Tabs
  defaultValue="cli"
  values={[
    { label: "vCluster CLI", value: "cli" },
    { label: "Helm", value: "helm" },
  ]}
>
  <TabItem value="cli">

<InterpolatedCodeBlock
  code={
`vcluster platform start \\
  --namespace vcluster-platform \\
  --kube-context [[VAR:FAILED_REGION_CONTEXT:arn:aws:eks:us-east-1:123456789012:cluster/platform-multi-region-us-east-1]] \\
  --values [[VAR:FAILED_REGION_VALUES_FILE:platform-us-east-1-values.yaml]] \\
  --upgrade \\
  --no-tunnel`
  }
  language="bash"
/>

  </TabItem>
  <TabItem value="helm">

<InterpolatedCodeBlock
  code={
`helm upgrade loft vcluster-platform --install --create-namespace --repository-config='' \\
  --namespace vcluster-platform \\
  --repo "https://charts.loft.sh/" \\
  --version [[VAR:CHART_VERSION:4.8.0]] \\
  --kube-context [[VAR:FAILED_REGION_CONTEXT:arn:aws:eks:us-east-1:123456789012:cluster/platform-multi-region-us-east-1]] \\
  -f [[VAR:FAILED_REGION_VALUES_FILE:platform-us-east-1-values.yaml]] \\
  --server-side=true --force-conflicts`
  }
  language="bash"
/>

  </TabItem>
</Tabs>

Then scale up if needed:

<InterpolatedCodeBlock
  code={
`kubectl --context [[VAR:FAILED_REGION_CONTEXT:arn:aws:eks:us-east-1:123456789012:cluster/platform-multi-region-us-east-1]] \\
  scale deployment -n vcluster-platform loft --replicas=[[VAR:REPLICA_COUNT:3]]`
  }
  language="bash"
/>

**If the EKS cluster itself is down**, follow the AWS documentation to restore
the cluster or create a replacement, then redeploy the platform using the
region's values file.

### Step 4 - Wait for the rollout to complete

<InterpolatedCodeBlock
  code={
`kubectl --context [[VAR:FAILED_REGION_CONTEXT:arn:aws:eks:us-east-1:123456789012:cluster/platform-multi-region-us-east-1]] \\
  rollout status deployment/loft -n vcluster-platform`
  }
  language="bash"
/>

### Step 5 - Verify recovery

Confirm the Route 53 health check for the restored region returns healthy:

<InterpolatedCodeBlock
  code={
`aws route53 get-health-check-status --health-check-id [[VAR:FAILED_REGION_HEALTH_CHECK_ID:xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx]] \\
  --query 'HealthCheckObservations[].{Region:Region,Status:StatusReport.Status}' \\
  --output table`
  }
  language="bash"
/>

:::note
Route 53 health checks run every 10 seconds with a failure threshold of 3.
After the platform becomes healthy, expect up to 30 seconds before the health
check status updates. DNS clients with cached responses may take an additional
60 seconds (the ALB alias record TTL) to start resolving to the restored region.
:::

Verify both regions are serving traffic:

<InterpolatedCodeBlock
  code={
`for CTX in [[GLOBAL:FIRST_REGION_CONTEXT]] [[GLOBAL:SECOND_REGION_CONTEXT]]; do
  echo "=== $CTX ==="
  kubectl --context "$CTX" get pods -n vcluster-platform -l app=loft
  echo
done`
  }
  language="bash"
/>

Confirm the custom DERP server is operational by checking for `NetworkPeer`
resources:

<InterpolatedCodeBlock
  code={
`kubectl --context [[VAR:FAILED_REGION_CONTEXT:arn:aws:eks:us-east-1:123456789012:cluster/platform-multi-region-us-east-1]] \\
  get networkpeers.storage.loft.sh -A`
  }
  language="bash"
/>

Both peers should show `Online: true`.
