# Create a vCluster template

## What are vCluster templates

vCluster templates are reusable configuration definitions that define how virtual Kubernetes clusters deploy in your environment. They contain standardized configurations including all necessary settings such as resource limits, security policies, and networking rules to create consistent vClusters across your organization.

When you create a template, you define a set of predetermined configurations that developers can use to deploy new vClusters without needing to understand complex Kubernetes YAML configurations.
- **Base configuration**: The core vCluster settings such as Kubernetes version, networking, and storage
- **Resource constraints**: CPU, memory, and storage limits that prevent resource overconsumption  
- **Governance rules**: Policies that control what users can and cannot modify when creating clusters

### Why templates matter for your organization

- **Standardization**: Every development team gets identical cluster configurations with the same security policies, resource limits, and networking rules. No more inconsistent environments between teams.

- **Self-service capability**: Developers create fully-configured vClusters through simple commands or UI clicks. They don't need to learn complex Kubernetes concepts or write YAML files.

- **Governance and control**: Platform teams define what settings users can change and what remains locked down. For example, developers might choose CPU limits within a range, but cannot disable security policies.

- **Tool integration**: Templates work with your existing tools including GitOps workflows, CI/CD pipelines, Helm charts, and Terraform infrastructure code.

- **Environment consistency**: Development, staging, and production environments use the same base template with different parameter values, eliminating "it works on my machine" problems.

## How to create templates

Templates use Kubernetes Custom Resource Definitions (CRDs), specifically the `VClusterTemplate` resource type. This means you manage templates using familiar tools such as kubectl, Helm, or Terraform.

The template creation process involves defining a YAML specification that describes your desired vCluster configuration. Once you apply this specification to your cluster, the template becomes available for users to create new vClusters.

### Create templates using kubectl

The kubectl method involves writing a YAML file that defines your template and applying it directly to your Kubernetes cluster. This gives you complete control over the template specification and integrates easily with GitOps workflows.

Use the following command to create a template using kubectl:

```bash
kubectl apply -f vclustertemplate.yaml
```

**Complete example template configuration:**

```yaml
apiVersion: vcluster.loft.sh/v1
kind: VClusterTemplate
metadata:
  name: development-template
  namespace: vcluster-templates
  labels:
    team: platform
    environment: development
spec:
  displayName: "Development environment template"
  description: "Standardized development vCluster with appropriate resource limits and development tools pre-installed"
  
  # Template defines the actual vCluster configuration
  template:
    metadata:
      labels:
        environment: development
        cost-center: engineering
        backup: enabled
    
    spec:
      # Control plane configuration
      controlPlane:
        distro:
          k8s:
            version: "1.28.4"  # Pin to specific Kubernetes version for consistency
        
        # Resource limits prevent clusters from consuming too many resources
        resources:
          limits:
            cpu: "2000m"      # Maximum 2 CPU cores
            memory: "4Gi"     # Maximum 4GB RAM
          requests:
            cpu: "200m"       # Guaranteed 200m CPU
            memory: "512Mi"   # Guaranteed 512MB RAM
        
        # Enable high availability for production-ready testing
        replicas: 2
      
      # Networking configuration
      networking:
        # Allow pods to resolve external DNS
        resolveDNS:
          - host.cluster.local
          - cluster.local
        
        # Enable ingress for web applications
        ingress:
          enabled: true
          className: nginx
      
      # Storage configuration  
      storage:
        persistence: true
        size: "10Gi"
        storageClass: "fast-ssd"
      
      # Security and access controls
      rbac:
        clusterRole:
          create: true
          rules:
            - apiGroups: [""]
              resources: ["pods", "services"]
              verbs: ["get", "list", "create", "update", "delete"]
      
      # Sync real resources from host cluster to vCluster
      sync:
        fromHost:
          nodes:
            enabled: true
        toHost:
          persistentVolumes:
            enabled: true
```

**Understanding the template structure:**

- **metadata section**: Contains the template name, namespace, and labels for organization
- **spec.displayName**: Human-readable name shown in user interfaces
- **spec.description**: Explains what this template provides and when to use it
- **spec.template**: Contains the actual vCluster configuration that gets applied when someone creates a cluster from this template

### Create templates using Helm

Helm provides a package management approach for templates, making it easier to distribute and version your templates across different environments. The Helm method works well when you need to deploy the same template with slight variations across multiple clusters.

First, add the official Loft Helm repository:

```bash
helm repo add loft https://charts.loft.sh
helm repo update
```

Then install your template using Helm:

```bash
helm install development-template loft/vcluster-template \
  --namespace vcluster-templates \
  --create-namespace \
  --set template.name=development-template \
  --set template.displayName="Development Environment Template" \
  --values template-values.yaml
```

**Example values file (template-values.yaml):**

```yaml
# Template metadata
template:
  metadata:
    name: development-template
    namespace: vcluster-templates
    labels:
      team: platform
      environment: dev

  # Template specification
  spec:
    displayName: "Development environment"
    description: "Pre-configured development environment with monitoring and debugging tools"
    
    template:
      spec:
        controlPlane:
          # Use lighter resource limits for development
          resources:
            limits:
              cpu: "1000m"
              memory: "2Gi" 
            requests:
              cpu: "100m"
              memory: "256Mi"
        
        # Development clusters don't need persistence by default
        storage:
          persistence: false
        
        # Enable useful development features
        networking:
          resolveDNS:
            - host.cluster.local
        
        # Add development-specific configurations
        env:
          - name: ENVIRONMENT
            value: "development"
          - name: LOG_LEVEL
            value: "debug"
```

**Benefits of the Helm approach:**

- **Version management**: Track template versions using Helm's built-in versioning
- **Environment promotion**: Use different values files for dev, staging, and production
- **Dependency management**: Include other Helm charts as dependencies
- **Rollback capability**: Easily rollback to previous template versions

### Create templates using Terraform

Terraform provides an Infrastructure as Code approach that integrates templates into your broader infrastructure management workflow. This method works best when you manage your entire Kubernetes infrastructure through Terraform.

**Basic Terraform template resource:**

```hcl
# Define the vCluster template resource
resource "kubernetes_manifest" "development_template" {
  manifest = {
    apiVersion = "vcluster.loft.sh/v1"
    kind       = "VClusterTemplate"
    
    metadata = {
      name      = "development-template"
      namespace = "vcluster-templates"
      labels = {
        team        = "platform"
        environment = "development"
        managed-by  = "terraform"
      }
    }
    
    spec = {
      displayName = "Development Environment Template"
      description = "Terraform-managed development template with standardized configurations"
      
      template = {
        metadata = {
          labels = {
            environment = "development"
            terraform   = "true"
          }
        }
        
        spec = {
          controlPlane = {
            distro = {
              k8s = {
                version = var.kubernetes_version
              }
            }
            
            resources = {
              limits = {
                cpu    = var.cpu_limit
                memory = var.memory_limit
              }
              requests = {
                cpu    = var.cpu_request
                memory = var.memory_request
              }
            }
          }
          
          storage = {
            persistence = var.enable_persistence
            size        = var.storage_size
          }
        }
      }
    }
  }
}

# Variables for customization
variable "kubernetes_version" {
  description = "Kubernetes version for the vCluster"
  type        = string
  default     = "1.28.4"
}

variable "cpu_limit" {
  description = "Maximum CPU allocation"
  type        = string
  default     = "1000m"
}

variable "memory_limit" {
  description = "Maximum memory allocation"
  type        = string
  default     = "2Gi"
}

variable "cpu_request" {
  description = "Guaranteed CPU allocation"
  type        = string
  default     = "100m"
}

variable "memory_request" {
  description = "Guaranteed memory allocation"
  type        = string
  default     = "256Mi"
}

variable "enable_persistence" {
  description = "Enable persistent storage"
  type        = bool
  default     = false
}

variable "storage_size" {
  description = "Storage size when persistence is enabled"
  type        = string
  default     = "10Gi"
}
```

**Advanced Terraform example with multiple templates:**

```hcl
# Create multiple templates for different environments
locals {
  templates = {
    development = {
      cpu_limit      = "1000m"
      memory_limit   = "2Gi"
      replicas       = 1
      persistence    = false
      storage_size   = "5Gi"
    }
    
    staging = {
      cpu_limit      = "2000m"
      memory_limit   = "4Gi"
      replicas       = 2
      persistence    = true
      storage_size   = "20Gi"
    }
    
    production = {
      cpu_limit      = "4000m"
      memory_limit   = "8Gi"
      replicas       = 3
      persistence    = true
      storage_size   = "50Gi"
    }
  }
}

# Generate templates for each environment
resource "kubernetes_manifest" "environment_templates" {
  for_each = local.templates
  
  manifest = {
    apiVersion = "vcluster.loft.sh/v1"
    kind       = "VClusterTemplate"
    
    metadata = {
      name      = "${each.key}-template"
      namespace = "vcluster-templates"
    }
    
    spec = {
      displayName = "${title(each.key)} Environment Template"
      description = "Template optimized for ${each.key} workloads"
      
      template = {
        spec = {
          controlPlane = {
            replicas = each.value.replicas
            resources = {
              limits = {
                cpu    = each.value.cpu_limit
                memory = each.value.memory_limit
              }
            }
          }
          
          storage = {
            persistence = each.value.persistence
            size        = each.value.storage_size
          }
        }
      }
    }
  }
}
```

## Template versioning capabilities

**Note: Template versioning requires Loft Pro**

Template versioning provides a way to manage template evolution over time. When you have versioning enabled, each change to a template creates a new version that you can track, reference, and rollback to if needed.

### How versioning works

Versioning operates automatically once you enable it:

1. **Initial creation**: When you first create a template, it exists without an explicit version number
2. **First update**: The moment you modify the template (through any method), the system creates version 1.0.0 
3. **Subsequent updates**: Each additional change increments the version number following semantic versioning principles
4. **Version tracking**: The system maintains a complete history of all template versions with their configurations and metadata

### Version management operations

**List all versions of a template:**

```bash
# Show version history for a specific template
loft list templates development-template --show-versions

# Output example:
# NAME                  VERSION   CREATED             MODIFIED
# development-template  1.2.3     2024-01-15 10:30    2024-02-20 14:45
# development-template  1.2.2     2024-01-10 09:15    2024-01-15 10:30
# development-template  1.2.1     2024-01-05 16:20    2024-01-10 09:15
```

**Deploy a vCluster using a specific template version:**

```bash
# Create vCluster with latest template version (default behavior)
loft create vcluster my-development-cluster --template development-template

# Create vCluster with specific template version
loft create vcluster my-stable-cluster \
  --template development-template \
  --template-version 1.2.2
```

**Compare template versions:**

```bash
# View differences between template versions
loft diff template development-template --from-version 1.2.2 --to-version 1.2.3
```

### Version rollback scenarios

Template versioning becomes crucial when you need to rollback problematic changes:

**Scenario 1: Problematic template update**
```bash
# Discover that version 1.3.0 has issues
# Rollback template to previous stable version
loft update template development-template --version 1.2.3

# All new vClusters now use the rolled-back configuration
# Existing vClusters remain unaffected unless explicitly updated
```

**Scenario 2: Gradual rollout**
```bash
# Deploy some clusters with new version for testing
loft create vcluster test-cluster --template development-template --template-version 1.3.0

# Keep production clusters on stable version
loft create vcluster prod-cluster --template development-template --template-version 1.2.3
```

## Template parameters for customization

Template parameters provide a mechanism for creating flexible templates that users can customize within predefined boundaries. Parameters let you expose specific configuration options while keeping other settings locked down for compliance and standardization.

### Understanding parameter types and validation

Parameters support various data types with complete validation options:

```yaml
apiVersion: vcluster.loft.sh/v1
kind: VClusterTemplate
metadata:
  name: configurable-template
spec:
  displayName: "Configurable development template"
  description: "Development template with user-customizable options"
  
  # Parameter definitions
  parameters:
    # String parameter with enumerated values
    - name: environment
      type: string
      description: "Target deployment environment - affects resource allocation and security policies"
      default: "development"
      required: true
      validation:
        enum: ["development", "staging", "production"]
        enumDescriptions:
          development: "Lightweight configuration for active development"
          staging: "Production-ready environment for testing"
          production: "Full production configuration with high availability"
    
    # String parameter with pattern validation
    - name: project_name
      type: string
      description: "Project identifier - must follow naming conventions"
      required: true
      validation:
        pattern: "^[a-z][a-z0-9-]*[a-z0-9]$"
        minLength: 3
        maxLength: 20
        patternDescription: "Lowercase letters, numbers, and hyphens only. Must start with letter."
    
    # Integer parameter with range validation
    - name: replica_count
      type: integer
      description: "Number of control plane replicas for high availability"
      default: 1
      validation:
        minimum: 1
        maximum: 5
        exclusiveMinimum: false
        exclusiveMaximum: false
    
    # String parameter for resource allocation
    - name: cpu_limit
      type: string
      description: "Maximum CPU allocation in millicores (such as 500m for half a core)"
      default: "1000m"
      validation:
        pattern: "^[0-9]+m$"
        patternDescription: "Must be in millicores format (such as 500m, 1000m, 2000m)"
    
    # Boolean parameter for feature toggles
    - name: enable_monitoring
      type: boolean
      description: "Install monitoring stack (Prometheus, Grafana) in the vCluster"
      default: true
    
    # Array parameter for multiple selections
    - name: enabled_features
      type: array
      description: "Additional features to enable in the cluster"
      default: ["ingress", "storage"]
      validation:
        items:
          type: string
          enum: ["ingress", "storage", "monitoring", "logging", "backup"]
        uniqueItems: true
        minItems: 1
        maxItems: 5

  # Template uses parameters through Go templating syntax
  template:
    metadata:
      name: "{{ .Values.project_name }}-cluster"
      labels:
        environment: "{{ .Values.environment }}"
        project: "{{ .Values.project_name }}"
        monitoring: "{{ .Values.enable_monitoring }}"
    
    spec:
      controlPlane:
        replicas: {{ .Values.replica_count }}
        
        resources:
          limits:
            cpu: "{{ .Values.cpu_limit }}"
            # Conditional logic based on environment
            memory: >-
              {{- if eq .Values.environment "production" -}}
              8Gi
              {{- else if eq .Values.environment "staging" -}}
              4Gi
              {{- else -}}
              2Gi
              {{- end }}
      
      # Conditional features based on array parameter
      {{- if has "monitoring" .Values.enabled_features }}
      monitoring:
        enabled: true
        prometheus:
          enabled: true
        grafana:
          enabled: true
      {{- end }}
      
      {{- if has "ingress" .Values.enabled_features }}
      networking:
        ingress:
          enabled: true
          className: nginx
      {{- end }}
```

### Using parameters when creating vClusters

Once you define a parameterized template, users can provide parameter values when creating vClusters:

**CLI method with individual parameters:**

```bash
# Create vCluster with custom parameters
loft create vcluster frontend-dev-cluster \
  --template configurable-template \
  --set environment=development \
  --set project_name=frontend-app \
  --set replica_count=2 \
  --set cpu_limit=1500m \
  --set enable_monitoring=true \
  --set enabled_features=ingress,storage,monitoring
```

**CLI method with values file:**

```bash
# Create values file
cat > cluster-values.yaml << EOF
environment: staging
project_name: backend-api
replica_count: 3
cpu_limit: 2000m
enable_monitoring: true
enabled_features:
  - ingress
  - storage
  - monitoring
  - logging
EOF

# Create vCluster using values file
loft create vcluster backend-staging-cluster \
  --template configurable-template \
  --values cluster-values.yaml
```

**Kubernetes manifest method:**

```yaml
apiVersion: vcluster.loft.sh/v1
kind: VCluster
metadata:
  name: production-cluster
  namespace: team-alpha
spec:
  templateRef:
    name: configurable-template
    # Optional: specify template version
    version: "1.2.3"
  
  # Parameter values
  parameters:
    environment: "production"
    project_name: "core-platform"
    replica_count: 5
    cpu_limit: "4000m"
    enable_monitoring: true
    enabled_features:
      - "ingress"
      - "storage" 
      - "monitoring"
      - "logging"
      - "backup"
```

### Advanced parameter features

**Conditional parameter visibility:**

```yaml
parameters:
  - name: environment
    type: string
    enum: ["development", "production"]
  
  # This parameter only appears when environment is production
  - name: backup_retention_days
    type: integer
    default: 30
    condition: "{{ eq .Values.environment 'production' }}"
    validation:
      minimum: 7
      maximum: 365
```

**Parameter dependencies:**

```yaml
parameters:
  - name: enable_database
    type: boolean
    default: false
  
  # Database configuration only required when database is enabled
  - name: database_size
    type: string
    required: true
    condition: "{{ .Values.enable_database }}"
    validation:
      enum: ["small", "medium", "large"]
  
  - name: database_backup_schedule
    type: string
    default: "0 2 * * *"
    condition: "{{ .Values.enable_database }}"
    validation:
      pattern: "^[0-9*/ ,-]+$"
      patternDescription: "Valid cron expression"
```

## Template validation and schema enforcement

Templates support complete JSON Schema validation to ensure parameter values meet your requirements and to prevent configuration errors before cluster creation.

### Schema validation structure

JSON Schema validation operates at multiple levels:

```yaml
apiVersion: vcluster.loft.sh/v1
kind: VClusterTemplate
metadata:
  name: enterprise-template
spec:
  # Parameter-level validation (shown above)
  parameters:
    - name: team_name
      type: string
      validation:
        pattern: "^(frontend|backend|platform|security)$"
  
  # Template-level validation schema
  jsonSchema:
    type: object
    required: ["team_name", "cost_center"]
    properties:
      team_name:
        type: string
        description: "Team responsible for this cluster"
        enum: ["frontend", "backend", "platform", "security"]
      
      cost_center:
        type: string
        description: "Billing cost center code"
        pattern: "^CC-[0-9]{4}$"
        examples: ["CC-1001", "CC-2003"]
      
      resource_tier:
        type: string
        description: "Resource allocation tier"
        enum: ["small", "medium", "large", "xlarge"]
        default: "medium"
      
      # Complex validation for resource requests
      custom_resources:
        type: object
        properties:
          cpu:
            type: string
            pattern: "^[0-9]+m$"
            description: "CPU in millicores"
          memory:
            type: string  
            pattern: "^[0-9]+[GMK]i$"
            description: "Memory with unit (Gi, Mi, Ki)"
        additionalProperties: false
        dependencies:
          cpu: ["memory"]
          memory: ["cpu"]
    
    # Conditional validation based on environment
    if:
      properties:
        environment:
          const: "production"
    then:
      required: ["backup_policy", "monitoring_config"]
      properties:
        backup_policy:
          type: object
          required: ["enabled", "retention_days"]
        monitoring_config:
          type: object
          required: ["alerts_enabled", "dashboard_enabled"]
```

### Custom validation functions

For complex business logic validation, you can define custom validation functions:

```yaml
spec:
  parameters:
    - name: cluster_size
      type: string
      validation:
        # Custom validation function
        customValidation: |
          function validate(value, allValues) {
            const environment = allValues.environment;
            const allowedSizes = {
              'development': ['small', 'medium'],
              'staging': ['medium', 'large'], 
              'production': ['large', 'xlarge']
            };
            
            if (!allowedSizes[environment].includes(value)) {
              return {
                valid: false,
                message: `Size '${value}' not allowed for environment '${environment}'. Allowed sizes: ${allowedSizes[environment].join(', ')}`
              };
            }
            
            return { valid: true };
          }
```
