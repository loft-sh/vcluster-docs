# Templates

vCluster templates define reusable configurations for creating virtual Kubernetes clusters. They allow platform teams to standardize cluster creation while giving users a controlled way to customize certain settings. Templates are implemented using Kubernetes Custom Resources (`VirtualClusterTemplate`) and can be managed with tools like `kubectl`, Helm, or Terraform.

Each template includes the base configuration for a virtual cluster—such as the Kubernetes version, control plane type, networking mode, and storage settings. Templates can also define resource constraints (CPU, memory, storage) to prevent overconsumption and ensure predictable usage.

To support secure and consistent self-service, templates allow administrators to define which settings users can modify through parameters. For example, you can expose only a few fields—like the number of replicas or a set of approved storage classes—while keeping the rest of the configuration fixed. This helps reduce the risk of misconfiguration while still enabling flexibility.

Templates can also include preinstalled applications, such as monitoring agents or ingress controllers, using Helm charts or Kubernetes manifests. Additional Kubernetes resources like `ConfigMaps`, `RBAC`, or `NetworkPolicies` can be applied to either the host cluster or the virtual cluster itself.

Because templates are versioned, they support upgrade workflows. When a template is updated, those changes can be propagated to any virtual clusters based on it—allowing teams to update cluster configuration and installed applications consistently across environments.


## Template types

vCluster Platform supports multiple types of templates. Each one serves a different purpose in the cluster lifecycle:

| Template Type.               | Custom Resource                                           | Purpose                                                                                                                   | Common Use Cases                                                                     |
| ---------------------------- | --------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| App Template.                | `AppTemplate`                                             | Defines reusable application configurations, such as Helm charts or container specifications.                             | Deploy standard applications across projects or spaces using consistent settings.        |
| Namespace Template.          |  `NamespaceTemplate`.                                     | Predefines configurations for new namespaces, including role-based access control, resource quotas, and labels.           | Automatically apply baseline policies when a namespace is created.                   |
| Virtual Cluster Template.    | `VirtualClusterTemplate`                                  | Specifies how virtual clusters are provisioned, including control plane settings, synchronization behavior, and metadata. | Provision virtual clusters with consistent architecture, features, and default settings. |


## Create a template

Before applying a template, complete the following setup steps:

1. Log in to the vCluster Platform:

```bash
vcluster platform login <PLATFORM_URL>
```

2. Connect your kubeconfig to the platform management API:

```bash
vcluster platform connect management
```

After connection, you can apply a template using `kubectl`, Helm, or Terraform.

You can also apply a `VirtualClusterTemplate` manifest directly to your cluster using the following command:

```bash
kubectl apply -f virtualclustertemplate.yaml
```

**Example manifest:**

```yaml
apiVersion: management.loft.sh/v1
kind: VirtualClusterTemplate
metadata:
  name: development-template
  labels:
    team: platform
    environment: development
spec:
  displayName: "Development environment template"
  description: "Standardized development vCluster with appropriate resource limits and development tools pre-installed"

  template:
    metadata:
      labels:
        environment: development
        cost-center: engineering
        backup: enabled

    helmRelease:
      chart:
        version: 0.25.0
      values: |
        controlPlane:
          distro:
            k8s:
              version: "1.28.4"
              resources:
                limits:
                  cpu: "2000m"
                  memory: "4Gi"
                requests:
                  cpu: "200m"
                  memory: "512Mi"

          statefulSet:
            highAvailability:
              replicas: 3
            persistence:
              volumeClaim:
                enabled: true
                size: "10Gi"
                storageClass: "fast-ssd"

        networking:
          resolveDNS:
            - hostname: host.cluster.local
              target:
                hostname: cluster.local
          ingress:
            enabled: true

        rbac:
          clusterRole:
            enabled: true
            extraRules:
              - apiGroups: [""]
                resources: ["pods", "services"]
                verbs: ["get", "list", "create", "update", "delete"]

        sync:
          fromHost:
            nodes:
              enabled: true
          toHost:
            persistentVolumes:
              enabled: true
```

<Flow id="create-space-template-ui">
  <Step>
    Select the <NavStep>Templates</NavStep> field on the left-most menu bar.
  </Step>
  <Step>

    <b>[Optional]</b> Update the template with any additional desired configuration options.
    <Flow id="create-space-template-optional-ui">
      <Step>
        The <Label>Definition</Label> tab contains settings relating to the space deployment itself, primarily related to sleeping.
      </Step>
      <Step>
        <Label>Objects</Label> is where you can define Kubernetes manifests that should be created in the virtual cluster.
      </Step>
      <Step>
        <Label>Apps</Label> allows you to select any pre-defined applications that should be installed into the virtual cluster.

      <Step>

      <Step>

## Apply parameters and versioning

Templates support optional parameterization and versioning:

- [Parameters](/platform/next/administer/templates/parameters): Enable dynamic customization of template fields, allowing users to specify values for CPU limits, labels, environment types, and other configuration options during vCluster provisioning.

- [Versioning](/platform/next/administer/templates/versioning): Enables you to manage and iterate template configurations across different releases, providing version control for your cluster templates.
