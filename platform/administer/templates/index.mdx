# Templates

vCluster templates are reusable configuration definitions that define how virtual Kubernetes clusters are deployed in your environment. Each template defines a consistent, repeatable setup that enables platform teams to enforce standards while allowing developers to self-serve clusters.

Templates are defined using Kubernetes Custom Resource Definitions (CRDs), specifically the `VirtualClusterTemplate` resource type. You manage them using standard tools such as `kubectl`, Helm, or Terraform.

Templates can include:

- **Base configuration**: Core settings for the vCluster such as Kubernetes version, control plane type, networking, and storage.
- **Resource constraints**: CPU, memory, and storage limits that prevent resource overconsumption.
- **Governance rules**: Policies that define what users can and cannot configure.
- **Preinstalled applications**: Helm-based Apps that are deployed automatically when a vCluster starts.
- **Additional Kubernetes resources**: Configurations like NetworkPolicies, ConfigMaps, and RBAC that are applied into the vCluster or the host.

Templates also integrate with existing tools such as GitOps workflows, CI/CD pipelines, Helm charts, and Terraform infrastructure code. Development, staging, and production environments can use the same base template with different parameters, ensuring consistency across teams. Templates also support upgrading the vCluster configuration and installed Apps across multiple virtual clusters.

## Use cases

Templates are critical to managing virtual cluster sprawl, governance, and developer productivity at scale. They give platform teams the ability to define consistent configurations, enforce guardrails, and enable repeatable automation across environments.

- **Standardization**: Every virtual cluster created from a template inherits consistent base configurations, such as Kubernetes version, storage classes, and networking behavior. This reduces variability and ensures all teams are working within approved boundaries.

- **Self-service provisioning**: Developers can create fully functional, secure, and production-compliant virtual clusters without needing deep Kubernetes expertise. Templates abstract away complexity, allowing faster onboarding and fewer support escalations.

- **Governance and control**: Platform administrators can lock down sensitive configurations while exposing only safe, parameterized inputs. This ensures inexperienced users cannot create insecure or over-resourced clusters.

- **Tool integration**: Templates work seamlessly with GitOps workflows, CI/CD pipelines, Helm, and Terraform. They can be stored, versioned, and automated like any other infrastructure component.

- **Environment alignment**: Use the same base template across development, staging, and production environments, with adjustable parameters for resource size or features. This eliminates issues caused by environment drift.

- **Automatic App installation**: Templates support preloading Helm-based applications or Kubernetes manifests. When a vCluster is created, tools like monitoring agents, ingress controllers, or cert managers are installed automatically.

- **Upgrade propagation**: When a template is updated, those changes can be applied across all virtual clusters using it—allowing bulk upgrades of both configuration and Apps with minimal manual work.

- **Access guardrails**: Templates act as a safeguard for provisioning. For example, a junior developer cannot accidentally expose a cluster to the internet or request excessive CPU if the template restricts these settings.


## Template types

vCluster Platform supports multiple types of templates. Each one serves a different purpose in the cluster lifecycle:

| Template Type.               | Custom Resource                                           | Purpose                                                                                                                   | Common Use Cases                                                                     |
| ---------------------------- | --------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| App Template.                | `AppTemplate`                                             | Defines reusable application configurations, such as Helm charts or container specifications.                             | Deploy standard applications across projects or spaces using consistent settings.        |
| Namespace Template.          |  `NamespaceTemplate`.                                     | Predefines configurations for new namespaces, including role-based access control, resource quotas, and labels.           | Automatically apply baseline policies when a namespace is created.                   |
| Virtual Cluster Template.    | `VirtualClusterTemplate`                                  | Specifies how virtual clusters are provisioned, including control plane settings, synchronization behavior, and metadata. | Provision virtual clusters with consistent architecture, features, and default settings. |



## Create a template

Before applying a template, complete the following setup steps:

1. Log in to the vCluster Platform:

```bash
vcluster platform login <PLATFORM_URL>
```

2. Connect your kubeconfig to the platform management API:

```bash
vcluster platform connect management
```

After connection, you can apply a template using `kubectl`, Helm, or Terraform.

You can also apply a `VirtualClusterTemplate` manifest directly to your cluster using the following command:

```bash
kubectl apply -f virtualclustertemplate.yaml
```

**Example manifest:**

```yaml
apiVersion: management.loft.sh/v1
kind: VirtualClusterTemplate
metadata:
  name: development-template
  labels:
    team: platform
    environment: development
spec:
  displayName: "Development environment template"
  description: "Standardized development vCluster with appropriate resource limits and development tools pre-installed"

  template:
    metadata:
      labels:
        environment: development
        cost-center: engineering
        backup: enabled

    helmRelease:
      chart:
        version: 0.25.0
      values: |
        controlPlane:
          distro:
            k8s:
              version: "1.28.4"
              resources:
                limits:
                  cpu: "2000m"
                  memory: "4Gi"
                requests:
                  cpu: "200m"
                  memory: "512Mi"

          statefulSet:
            highAvailability:
              replicas: 3
            persistence:
              volumeClaim:
                enabled: true
                size: "10Gi"
                storageClass: "fast-ssd"

        networking:
          resolveDNS:
            - hostname: host.cluster.local
              target:
                hostname: cluster.local
          ingress:
            enabled: true

        rbac:
          clusterRole:
            enabled: true
            extraRules:
              - apiGroups: [""]
                resources: ["pods", "services"]
                verbs: ["get", "list", "create", "update", "delete"]

        sync:
          fromHost:
            nodes:
              enabled: true
          toHost:
            persistentVolumes:
              enabled: true
```

## Apply parameters and versioning

Templates support optional parameterization and versioning:

- [Parameters](/platform/administer/templates/parameters): Enable dynamic customization of template fields, allowing users to specify values for CPU limits, labels, environment types, and other configuration options during vCluster provisioning.

- [Versioning](/platform/administer/templates/versioning): Enables you to manage and iterate template configurations across different releases, providing version control for your cluster templates.
