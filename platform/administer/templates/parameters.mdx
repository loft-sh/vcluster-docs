# Parameters

Template parameters provide a way to create flexible, reusable vCluster templates while maintaining control over what users can modify. Parameters expose specific configuration options to end users during virtual cluster creation, allowing customization within predefined boundaries set by administrators.

When you create a vCluster from a parameterized template, the platform presents a form where users enter values for each parameter. These values are then injected into the template using Go template syntax, dynamically customizing the virtual cluster configuration.

- **Controlled flexibility**: Users can customize specific aspects of their virtual clusters without needing to understand complex YAML configurations
- **Standardization**: Core settings remain consistent across all deployments while allowing variation where appropriate
- **Self-service enablement**: Developers can create customized virtual clusters without requiring administrator intervention
- **Governance compliance**: Administrators control what can be changed and limit choices to approved options

### How parameters work

Parameters work through a simple templating process:

1. **Parameter definition**: Administrators define parameters in the template, specifying types, validation rules, and allowed values
2. **User input**: When creating a vCluster, users see a form with the defined parameters and provide their values
3. **Template rendering**: The platform injects user values into the template using Go template syntax such as `{{ .Values.parameterName }}`
4. **vCluster creation**: The rendered template creates a customized virtual cluster with the user's specified configuration

## Parameter types and validation

### Supported parameter types

| Type | Description | Use cases |
|------|-------------|-----------|
| `String` | Text input field | Names, labels, configuration values |
| `Number` | Numeric input with validation | Replica counts, port numbers, resource limits |
| `Boolean` | Checkbox for true/false values | Feature toggles, enable/disable settings |
| `Password` | Masked text input | Passwords, API keys, secrets |
| `Multiline` | Large text area | Configuration files, scripts, descriptions |

### Validation options

Parameters support multiple validation mechanisms:

- **Required fields**: Mark parameters as mandatory
- **Allowed options**: Restrict input to predefined choices
- **Regular expressions**: Validate format for free-form text
- **Numeric ranges**: Set minimum and maximum values for numbers

## Define parameters using the UI

The platform UI provides an intuitive interface for creating and managing template parameters.

<Flow id="ui-template-parameters">
  <Step>
  Navigate to your template in the platform UI and click **Add parameter** in the template editor.
  </Step>
  <Step>
  From the **Section** dropdown, choose an existing section or create a new one (such as "Environment Settings" or "Resource Configuration"). Sections help organize related parameters and improve the user experience during deployment.
  </Step>
  <Step>
  Enter a unique **Identifier (variable)** name. This becomes the variable key used in your template (for example, `cpu_limit` becomes `{{ .Values.cpu_limit }}`). Use descriptive names that clearly indicate the parameter's purpose.
  </Step>
  <Step>
  Select the appropriate **Type** for your parameter from the dropdown menu. Consider what kind of input makes most sense for your use case.
  </Step>
  <Step>
  Provide a clear **Label** that end users will see in the deployment form. Make it descriptive and user-friendly (for example, "CPU Limit" instead of "cpu_limit").
  </Step>
  <Step>
  Add a comprehensive **Description** that explains what the parameter controls and provides guidance on appropriate values. Include examples when helpful.
  </Step>
  <Step>
  Toggle **Required** if users must provide a value for this parameter before deploying. Consider whether a default value might be more appropriate than making it required.
  </Step>
  <Step>
  If you want to restrict user choices, use **Allowed Options** to create a dropdown menu. Click **+ Add option** for each allowed value. This is particularly useful for environment names, regions, or configuration presets.
  </Step>
  <Step>
  Click **Done** to save the parameter and add it to your template.
  </Step>
</Flow>

### Parameter organization best practices

- **Group related parameters**: Use sections to organize parameters logically, such as "Resource Settings", "Network Configuration", or "Security Options"
- **Order by importance**: Place the most commonly modified parameters first
- **Use clear naming**: Choose parameter identifiers that are both human-readable and meaningful in your templates
- **Provide helpful descriptions**: Include guidance, examples, and constraints in parameter descriptions

### Advanced parameter settings

Access additional configuration options through the **Advanced** menu:

- **Default values**: Set sensible defaults to reduce user input requirements
- **Conditional display**: Show parameters only when specific conditions are met
- **Dependencies**: Link parameters so changing one affects the options available in another
- **Custom validation**: Apply complex validation rules beyond basic types

## YAML parameter definition

For templates managed through Git or automated workflows, you can define parameters directly in YAML format. This approach provides more control and enables version control of your parameter definitions.

### Basic parameter structure

```yaml
parameters:
  - variable: environment_type
    label: Environment Type
    description: Select the target environment for this virtual cluster
    type: string
    required: true
    options:
      - development
      - staging
      - production
    section: Environment Settings
```

### Parameter examples by type

#### String parameter with validation

```yaml
- variable: cluster_name
  label: Cluster Name
  description: Enter a name for your virtual cluster (3-20 characters, lowercase letters and hyphens only)
  type: string
  required: true
  validation: "^[a-z][a-z0-9-]*[a-z0-9]$"
  section: Basic Configuration
```

#### Number parameter with range

```yaml
- variable: replica_count
  label: Replica Count
  description: Number of application replicas (1-10)
  type: number
  required: true
  min: 1
  max: 10
  default: 3
  section: Scaling Settings
```

#### Boolean parameter for feature toggle

```yaml
- variable: enable_monitoring
  label: Enable Monitoring
  description: Install Prometheus and Grafana monitoring stack
  type: boolean
  default: true
  section: Optional Features
```

#### Password parameter for sensitive data

```yaml
- variable: admin_password
  label: Admin Password
  description: Password for the admin user (minimum 8 characters)
  type: password
  required: true
  validation: "^.{8,}$"
  section: Security Settings
```

#### Multiline parameter for configuration

```yaml
- variable: custom_config
  label: Custom Configuration
  description: Additional YAML configuration to apply to the virtual cluster
  type: multiline
  section: Advanced Settings
```

### Complex parameter validation

#### Enum with descriptions

```yaml
- variable: instance_size
  label: Instance Size
  description: Choose the resource allocation tier for your virtual cluster
  type: string
  required: true
  options:
    - value: small
      description: 1 CPU, 2GB RAM - suitable for development
    - value: medium
      description: 2 CPU, 4GB RAM - suitable for testing
    - value: large
      description: 4 CPU, 8GB RAM - suitable for production workloads
  section: Resource Configuration
```

#### Conditional parameters

```yaml
- variable: enable_persistence
  label: Enable Persistent Storage
  description: Add persistent volumes to the virtual cluster
  type: boolean
  default: false
  section: Storage Settings

- variable: storage_size
  label: Storage Size
  description: Size of persistent storage (e.g., 10Gi, 50Gi)
  type: string
  required: true
  validation: "^[0-9]+[KMGT]i$"
  condition: "{{ .Values.enable_persistence }}"
  section: Storage Settings
```

## Access parameter values in templates

Once you define parameters, access their values in your template using Go template syntax. The platform makes all parameter values available under the `.Values` object.

### Basic parameter access

```yaml
apiVersion: vcluster.loft.sh/v1
kind: VCluster
metadata:
  name: "{{ .Values.cluster_name }}"
  labels:
    environment: "{{ .Values.environment_type }}"
    project: "{{ .Values.project_name }}"
spec:
  controlPlane:
    replicas: {{ .Values.replica_count }}
```

### Conditional template logic

Use Go template conditionals to create dynamic configurations based on parameter values:

```yaml
spec:
  controlPlane:
    resources:
      limits:
        # Different resource limits based on environment
        cpu: >-
          {{- if eq .Values.environment_type "production" -}}
          4000m
          {{- else if eq .Values.environment_type "staging" -}}
          2000m
          {{- else -}}
          1000m
          {{- end }}
        memory: >-
          {{- if eq .Values.environment_type "production" -}}
          8Gi
          {{- else if eq .Values.environment_type "staging" -}}
          4Gi
          {{- else -}}
          2Gi
          {{- end }}
  
  # Conditional features based on boolean parameters
  {{- if .Values.enable_monitoring }}
  monitoring:
    enabled: true
    prometheus:
      enabled: true
    grafana:
      enabled: true
  {{- end }}
  
  {{- if .Values.enable_persistence }}
  storage:
    persistence: true
    size: "{{ .Values.storage_size }}"
  {{- end }}
```

### String manipulation and defaults

Use Go template functions for string manipulation and providing fallback values:

```yaml
metadata:
  name: "{{ .Values.cluster_name | lower }}"
  labels:
    # Use default value if parameter is empty
    cost-center: "{{ .Values.cost_center | default "engineering" }}"
    # Combine multiple parameters
    full-name: "{{ .Values.project_name }}-{{ .Values.environment_type }}"
    # Transform string values
    normalized-name: "{{ .Values.cluster_name | lower | replace "_" "-" }}"
```

### Working with complex parameter types

#### Array parameters

```yaml
# Parameter definition
- variable: enabled_features
  label: Enabled Features
  description: Select features to enable
  type: array
  options:
    - ingress
    - monitoring
    - logging
    - backup
  section: Features

# Template usage
{{- range .Values.enabled_features }}
{{- if eq . "ingress" }}
networking:
  ingress:
    enabled: true
{{- end }}
{{- if eq . "monitoring" }}
monitoring:
  enabled: true
{{- end }}
{{- end }}
```

#### Object parameters

```yaml
# Access nested object values
metadata:
  labels:
    team: "{{ .Values.team_config.name }}"
    contact: "{{ .Values.team_config.email }}"
```

## Platform-specific parameter values

The platform automatically provides context-specific values that you can use in your templates alongside user-defined parameters. These built-in values help create dynamic configurations that adapt to the deployment environment.

### Available platform parameters

| Parameter | Description | Example Value |
|-----------|-------------|---------------|
| `.Values.loft.project` | Name of the project containing the virtual cluster | `frontend-team` |
| `.Values.loft.space` | Name of the space (for space-scoped resources) | `development-space` |
| `.Values.loft.virtualClusterName` | Name of the virtual cluster being created | `my-dev-cluster` |
| `.Values.loft.cluster` | Name of the physical host cluster | `production-k8s` |
| `.Values.loft.virtualClusterNamespace` | Kubernetes namespace for the virtual cluster | `vcluster-my-dev-cluster` |
| `.Values.loft.user.name` | Username of the person creating the cluster | `john.doe` |
| `.Values.loft.user.email` | Email address of the user | `john.doe@company.com` |
| `.Values.loft.team.name` | Team name (if created by a team) | `platform-engineering` |

### Using platform parameters

```yaml
apiVersion: vcluster.loft.sh/v1
kind: VCluster
metadata:
  name: "{{ .Values.loft.virtualClusterName }}"
  labels:
    # Automatically tag resources with context
    project: "{{ .Values.loft.project }}"
    cluster: "{{ .Values.loft.cluster }}"
    owner: "{{ .Values.loft.user.name }}"
    created-by: "vcluster-platform"
  annotations:
    contact: "{{ .Values.loft.user.email }}"
    
spec:
  # Use project name in resource naming
  controlPlane:
    statefulSet:
      metadata:
        labels:
          project: "{{ .Values.loft.project }}"
          environment: "{{ .Values.environment_type }}"
```

### Combining platform and user parameters

Create sophisticated naming and tagging strategies by combining platform context with user input:

```yaml
metadata:
  name: "{{ .Values.loft.project }}-{{ .Values.environment_type }}-{{ .Values.loft.user.name | replace "." "-" }}"
  labels:
    # Standard organizational labels
    project: "{{ .Values.loft.project }}"
    environment: "{{ .Values.environment_type }}"
    owner: "{{ .Values.loft.user.name }}"
    cluster: "{{ .Values.loft.cluster }}"
    
    # Custom user-defined labels
    cost-center: "{{ .Values.cost_center }}"
    application: "{{ .Values.application_name }}"
    
  annotations:
    # Useful metadata for operations
    created-by: "{{ .Values.loft.user.email }}"
    created-at: "{{ now | date "2006-01-02T15:04:05Z" }}"
    project-id: "{{ .Values.loft.project }}"
```

## Parameter validation and error handling

### Input validation strategies

#### Format validation with regular expressions

```yaml
- variable: kubernetes_version
  label: Kubernetes Version
  description: Kubernetes version (e.g., 1.28.4, 1.27.8)
  type: string
  required: true
  validation: "^1\.(2[4-9]|[3-9][0-9])\.[0-9]+$"
  section: Configuration
```

#### Range validation for numbers

```yaml
- variable: worker_nodes
  label: Worker Nodes
  description: Number of worker nodes (1-20)
  type: number
  required: true
  min: 1
  max: 20
  section: Scaling
```

#### Complex validation with multiple constraints

```yaml
- variable: storage_class
  label: Storage Class
  description: Kubernetes storage class for persistent volumes
  type: string
  required: true
  options:
    - fast-ssd
    - standard
    - archive
  validation: "^(fast-ssd|standard|archive)$"
  section: Storage
```

### Error handling in templates

Provide meaningful error messages and fallbacks in your templates:

```yaml
spec:
  controlPlane:
    # Validate required parameters
    {{- if not .Values.environment_type }}
    {{- fail "environment_type parameter is required" }}
    {{- end }}
    
    # Provide reasonable defaults
    replicas: {{ .Values.replica_count | default 1 }}
    
    resources:
      limits:
        # Ensure resource values are valid
        cpu: >-
          {{- if .Values.cpu_limit -}}
          {{ .Values.cpu_limit }}
          {{- else if eq .Values.environment_type "production" -}}
          2000m
          {{- else -}}
          1000m
          {{- end }}
```

## Best practices for template parameters

### Design principles

1. **Start simple**: Begin with essential parameters and add complexity as needed
2. **Provide sensible defaults**: Reduce required user input while maintaining flexibility
3. **Use descriptive names**: Make parameter purposes clear from their identifiers and labels
4. **Group logically**: Organize related parameters into sections for better user experience
5. **Validate input**: Use appropriate validation to prevent configuration errors

### Security considerations

- **Sensitive data**: Use password-type parameters for secrets and API keys
- **Input sanitization**: Validate user input to prevent injection attacks
- **Least privilege**: Only expose parameters that users actually need to modify
- **Audit trail**: Consider how parameter choices affect compliance and auditing requirements

### User experience guidelines

- **Clear descriptions**: Explain what each parameter does and provide examples
- **Logical ordering**: Place most important and commonly changed parameters first
- **Reasonable defaults**: Set defaults that work for most use cases
- **Helpful constraints**: Use allowed options to guide users toward valid choices

### Template maintenance

- **Version parameters**: Consider how parameter changes affect template versioning
- **Backward compatibility**: Plan for handling parameters when templates evolve
- **Documentation**: Maintain clear documentation of parameter purposes and valid values
- **Testing**: Test templates with various parameter combinations to ensure robustness

## Troubleshooting common parameter issues

### Template rendering errors

**Problem**: Template fails to render with parameter values
**Solution**: Check Go template syntax and ensure all referenced parameters exist

```yaml
# Incorrect - missing default for optional parameter
cpu: "{{ .Values.cpu_limit }}"

# Correct - provide default value
cpu: "{{ .Values.cpu_limit | default "1000m" }}"
```

### Parameter validation failures

**Problem**: Users cannot submit valid values
**Solution**: Review validation patterns and ensure they match expected input formats

```yaml
# Too restrictive pattern
validation: "^[a-z]{3}$"  # Only allows exactly 3 lowercase letters

# More flexible pattern
validation: "^[a-z][a-z0-9-]*[a-z0-9]$"  # Allows reasonable names
```

### Missing parameter values

**Problem**: Required parameters not being prompted
**Solution**: Verify parameter configuration and template syntax

```yaml
# Ensure parameter is properly defined
- variable: required_field
  label: Required Field
  type: string
  required: true  # This flag is crucial
```

### Parameter not appearing in UI

**Problem**: Defined parameters don't show in deployment form
**Solution**: Check parameter syntax and ensure template is saved correctly

```yaml
# Verify proper YAML structure
parameters:  # Must be at root level
  - variable: my_param
    label: My Parameter
    type: string
```
