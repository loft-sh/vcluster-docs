The quick starts prove that vCluster can run in your environment. This section maps the path from "it runs" to "we can operate it" -- organized by what you are building, not by which quick start you ran.

## What production-ready means

A production-ready vCluster setup delivers:

- **Tenant isolation:** every customer or team sees only their own cluster, nodes, and workloads
- **Repeatable provisioning:** new tenant clusters deploy from a defined template, not from manual steps
- **A defined worker node model:** shared nodes, dedicated node pools, private nodes, or Standalone -- matched to your security, performance, and cost requirements
- **Governed access:** who can create, access, and administer tenant clusters, enforced through Platform policies
- **Durable control planes:** HA, data store, and backup procedures defined before tenants depend on the system
- **Operational readiness:** monitoring, upgrade, restore, and incident response procedures documented and tested

## Reference architecture models

These models map to the use-case paths above and provide a shared vocabulary for architecture discussions.

| Model | Name | Summary |
| --- | --- | --- |
| 1 | Single-Tenant Stack | Whole platform stack deployed per customer. Each customer has their own Control Plane Cluster and private GPU nodes. |
| 2 | Multi-Tenant Platform | Operator-owned platform hidden from customers. Customer-facing provisioning goes through the operator's own API or portal, not Platform directly. |
| 3 | Capacity Aggregator | Central Platform (global) with per-DC Standalone Control Plane Clusters. One management plane, multiple GPU suppliers. |
| 4 | Edge Distribution | Central Control Plane Cluster, workers at distributed edge sites connected via VPN. |
| 5 | Enterprise AI Factory | Internal AI platform with production teams on private GPU nodes and dev/experiment teams on a shared node pool. |
