Build an internal AI platform for your organization's data science, R&D, and line-of-business teams. Production AI teams get tenant clusters with private GPU nodes. Experimental and dev teams share a node pool. Platform governs access, quotas, and lifecycle across all teams.

**Typical stack:** Platform on an existing cloud Kubernetes cluster or Standalone on-premises. Private nodes for production ML teams. Shared node pool for dev and experiment workloads. vMetal optional for on-premises GPU fleets.

**What makes this path different:** Teams interact with Platform directly (or through an internal portal). The focus is on self-service provisioning, standardized GPU environments, and chargeback visibility.

## Day 0: Design decisions

| Decision | Read next | Outcome |
| --- | --- | --- |
| Decide the Control Plane Cluster foundation | [Architecture](../introduction/architecture.mdx), [Deployment basics](../deploy/control-plane/kubernetes-pod/basics.mdx) | Run the Control Plane Cluster on an existing managed Kubernetes service (EKS, AKS, GKE) or vCluster Standalone on on-premises servers. |
| Define team tenancy tiers | [Private Nodes](../deploy/worker-nodes/private-nodes/overview.mdx), [Deployment basics](../deploy/control-plane/kubernetes-pod/basics.mdx) | Classify teams: production AI workloads on private GPU nodes, dev and experiment on shared nodes. |
| Define project and quota structure | [Projects](/platform/administer/projects/create), [Quotas](/platform/administer/projects/quotas) | Map projects to teams or business units. Set GPU, CPU, and memory quotas per project. |
| Plan GPU tooling and AI stacks | [Certified Stacks](/platform/integrations/certified-stacks) | Standardize GPU Operator, scheduler (Run.ai, Kueue, Volcano), and developer environment (Jupyter, VS Code) across teams using Certified Stacks. |
| Choose the runtime isolation model | [vNode docs](https://www.vnode.com/docs/) | vNode is recommended when teams run agentic workloads, untrusted code execution, or need root access inside containers without the risk of node escape. |
| Plan SSO and governance | [SSO](/platform/configure/platform-configs/single-sign-on), [RBAC](/platform/administer/users-permissions/overview) | Integrate with your corporate identity provider. Define which teams can self-service create clusters and which require approval. |

## Day 1: Stand up the first production environment

1. Install [vCluster Platform](/platform/install/quick-start-guide) on your Control Plane Cluster.
2. Configure [SSO](/platform/configure/platform-configs/single-sign-on), [teams, and permissions](/platform/administer/users-permissions/overview) against your corporate identity provider.
3. Create [projects](/platform/administer/projects/create), [templates](/platform/administer/templates/overview), and [quotas](/platform/administer/projects/quotas) per team or business unit.
4. Configure the shared node pool: [pod security standard](../configure/vcluster-yaml/policies/pod-security-standard.mdx), [network policy](../configure/vcluster-yaml/policies/network-policy.mdx), and [resource quota](../configure/vcluster-yaml/policies/resource-quota.mdx).
5. Configure private GPU nodes for production teams: [join nodes manually](../deploy/worker-nodes/private-nodes/join.mdx) or configure [Auto Nodes](/platform/administer/node-providers/overview).
6. Set up [vMetal](https://www.vmetal.ai/docs/) if on-premises GPU servers are part of the fleet.
7. Deploy [Certified Stacks](/platform/integrations/certified-stacks) as the baseline template for GPU-enabled tenant clusters.
8. Install [vNode](https://www.vnode.com/docs/) on nodes designated for agentic or privileged workloads.
9. Provision the first tenant cluster for each tier (shared and private) and validate GPU access, isolation, and quota enforcement.
10. Document how teams request clusters, retrieve kubeconfigs, request quota increases, and report issues.
