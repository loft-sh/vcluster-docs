<ProAdmonition/>

You can configure vCluster to sync CustomResources from the host cluster to the virtual cluster. vCluster reads these resources from the host cluster and creates read-only copies in the virtual cluster.

The sync process works in two steps:

1. vCluster copies the Custom Resource Definition (CRD) from the host cluster to the virtual cluster.
2. vCluster syncs the actual resources and maintains them as read-only in the virtual cluster.

This feature enables apps in vCluster to use CustomResources from the host cluster. Common use cases include accessing cluster-scoped resources like ClusterIssuers (for [cert-manager](https://cert-manager.io/)) or ClusterStores (for [external-secrets](https://external-secrets.io/latest/)), as well as syncing specific namespace-scoped resources.

vCluster provides three approaches for syncing CustomResources from the host to virtual cluster:

- [Cluster-scoped sync](#cluster-sync) - Sync cluster-wide resources like ClusterIssuers or ClusterStores.
- [Versioned sync](#versioned-sync) - Specify a particular API version when multiple versions exist.
- [Namespace-scoped sync](#namespace-sync) - Sync resources from specific host namespaces to designated virtual namespaces with granular mapping control.

:::note
The CustomResources feature syncs resources from the host to the virtual cluster. If you want to sync resources from the virtual cluster to the host, see [syncing custom resources to the host cluster](../to-host/advanced/custom-resources).
:::

:::info No need to configure RBAC
vCluster automatically adds the required RBAC permissions to retrieve CRDs and sync resources from the host to the virtual cluster.
:::

## Cluster-scoped sync {#cluster-sync}

When configuring vCluster to sync ClusterIssuers from the host cluster (from [cert-manager](https://cert-manager.io/)), you do not need to specify an API version by default. 
vCluster automatically uses the storage version when multiple API versions exist and you do not define a specific version. If your CRD has multiple versions and you need a specific API version, specify it in your `vcluster.yaml` configuration:

```yaml title="configure Cluster scoped CRD sync from host"
sync:
  fromHost:
    customResources:
      clusterissuers.cert-manager.io:
        enabled: true
        scope: Cluster
```

## Versioned sync {#versioned-sync}

To use a specific API version instead of the default storage version, add the version after the "/" separator in your configuration. This is useful when you need a particular version of a CRD that differs from the storage version, or when you are uncertain which version is the storage version.

For example, to specify version `v1` of the `clusterissuers.cert-manager.io` CRD:

```yaml title="configure Cluster scoped CRD sync from host with version"
sync:
  fromHost:
    customResources:
      clusterissuers.cert-manager.io/v1:
        enabled: true
        scope: Cluster
```

:::info Unique CRD versions
vCluster can only sync one version of a CustomResource at a time. If multiple versions are specified, vCluster fails to start.
:::

## Namespace-scoped sync {#namespace-sync}

You can configure vCluster to sync namespace-scoped CustomResources from specific namespaces in the host cluster to designated namespaces in the virtual cluster. This feature requires explicit configuration using namespace mappings to specify exactly which resources to sync and where to place them.

### Prerequisites

Before configuring namespace-scoped CustomResource sync, ensure the following requirements are met:

- All specified host namespaces must exist when vCluster starts.
- The CustomResourceDefinition must exist in the host cluster when vCluster starts.
- vCluster automatically creates the required RBAC permissions for retrieving CustomResourceDefinitions and syncing resources.

### How namespace mapping works

Namespace-scoped CustomResource sync requires you to specify `mappings.byName` in your configuration. This mapping tells vCluster exactly which host resources to sync and their destination in the virtual cluster. You use the format `"host-namespace/resource-name": "virtual-namespace/resource-name"` to define these mappings.

vCluster automatically creates target namespaces in the virtual cluster if they do not already exist. The sync operates in one direction only, from host to virtual cluster. When you modify a resource in the host cluster, vCluster syncs the change to the virtual cluster. When you delete a virtual resource, vCluster recreates it if the host resource still exists. When you delete a host resource, vCluster removes the corresponding virtual resource.

### Sync limitations

- You cannot sync from all namespaces in the host cluster; you must specify individual namespace mappings.
- You cannot sync CustomResources that were previously synced from virtual to host cluster; vCluster skips these resources.
- vCluster can only sync one version of a CustomResource at a time.

### Sync all resources from a namespace

To sync all CustomResources of a specific type from one host namespace to a virtual namespace, use the wildcard pattern `namespace/*`:

```yaml title="Sync all CertificateRequests from host namespace"
sync:
  fromHost:
    customResources:
      certificaterequests.cert-manager.io:
        enabled: true
        scope: Namespaced
        mappings:
          byName:
            # Syncs all CertificateRequests from "production" namespace in host
            # to "cert-manager" namespace in virtual cluster
            "production/*": "cert-manager/*"
```

This configuration syncs all CertificateRequest resources from the `production` namespace in the host cluster to the `cert-manager` namespace in the virtual cluster. Resource names remain unchanged during the sync process.

### Sync specific resources

To sync only specific CustomResources rather than all resources from a namespace, specify the exact resource name using the format `namespace/name`:

```yaml title="Sync specific CertificateRequest"
sync:
  fromHost:
    customResources:
      certificaterequests.cert-manager.io:
        enabled: true
        scope: Namespaced
        mappings:
          byName:
            # Syncs only the "web-cert" CertificateRequest from "production" host namespace
            # to "certificates" namespace in virtual cluster
            "production/web-cert": "certificates/web-cert"
            # Syncs only the "api-cert" CertificateRequest from "staging" host namespace
            # to "certificates" namespace in virtual cluster
            "staging/api-cert": "certificates/api-cert"
```

### Sync from vCluster's own host namespace

Since the namespace might not always be known in advance (especially when created by vCluster Platform), you can use an empty string `""` to represent vCluster's own host namespace:

```yaml title="Sync from vCluster's host namespace"
sync:
  fromHost:
    customResources:
      certificaterequests.cert-manager.io:
        enabled: true
        scope: Namespaced
        mappings:
          byName:
            # Syncs all CertificateRequests from vCluster's host namespace
            # to "certificates" namespace in virtual cluster
            "": "certificates/*"
```

You can also sync specific resources from vCluster's host namespace by using a slash:

```yaml title="Sync specific resource from vCluster's host namespace"
sync:
  fromHost:
    customResources:
      certificaterequests.cert-manager.io:
        enabled: true
        scope: Namespaced
        mappings:
          byName:
            # Syncs only "my-cert" CertificateRequest from vCluster's host namespace
            # to "certificates" namespace in virtual cluster
            "/my-cert": "certificates/my-cert"
```

### Modify names during sync

You can change both the namespace and name of resources during the sync process. This is useful for avoiding naming conflicts or organizing resources differently in the virtual cluster:

```yaml title="Modify resource names and namespaces during sync"
sync:
  fromHost:
    customResources:
      certificaterequests.cert-manager.io:
        enabled: true
        scope: Namespaced
        mappings:
          byName:
            # Syncs "web-cert" from "production" host namespace
            # as "prod-web-cert" in "certificates" virtual namespace
            "production/web-cert": "certificates/prod-web-cert"
            # Syncs "api-cert" from "staging" host namespace
            # as "stage-api-cert" in "certificates" virtual namespace
            "staging/api-cert": "certificates/stage-api-cert"
```

This configuration allows you to consolidate resources from multiple host namespaces into a single virtual namespace while maintaining clear naming to indicate their origin.

### Patches

You can apply patches to modify CustomResource content during the sync process using `reverseExpression` in the `sync.fromHost.customResources[*].patches` configuration. These patches operate on the host object and the modified values appear in the virtual object.

:::note Expression types for host-to-virtual sync
Only `reverseExpression` works for host-to-virtual sync. The `expression` field has no effect in this context. When syncing from host to virtual cluster, vCluster applies the `reverseExpression` to transform values from the host object before creating the virtual object.
:::

```yaml title="Apply patches during CustomResource sync"
sync:
  fromHost:
    customResources:
      certificaterequests.cert-manager.io:
        enabled: true
        scope: Namespaced
        mappings:
          byName:
            "production/web-cert": "certificates/prod-web-cert"
        patches:
          - path: metadata.annotations[*]
            # Remove 'www.' prefix from annotation values during sync
            reverseExpression: "value.startsWith('www.') ? value.slice(4) : value"
```
