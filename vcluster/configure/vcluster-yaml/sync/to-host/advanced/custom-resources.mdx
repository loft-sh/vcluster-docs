<ProAdmonition/>

vCluster allows you to sync CustomResources from the virtual cluster to the host cluster. You can sync arbitrary resources that vCluster does not sync by default. This feature only works for resources that have a CustomResourceDefinition in the host cluster.

vCluster automatically syncs any resources that CustomResources create in the host cluster back to the virtual cluster. For example, a [cert-manager](https://cert-manager.io/) Certificate creates a Secret in the host cluster, which vCluster then syncs back to the virtual cluster automatically.

vCluster automatically adds the necessary RBAC permissions for retrieving CustomResourceDefinitions and syncing resources between clusters.
<br />

:::info Only Namespace-Scoped Resource
This feature currently only works for namespace-scoped resources only.
:::

:::info Multi-Namespace-Mode
If you want to sync many custom resources, consider using [namespace syncing](../../../sync/to-host/advanced/namespaces).
:::

## Enable custom resource syncing {#enable-custom-resource-syncing}

To enable CustomResource syncing from the virtual cluster to the host cluster, first identify which CustomResourceDefinitions you want to sync by running `kubectl get crds`. Add the CustomResourceDefinition names to the `customResources` section in the sync configuration. 

When you sync CustomResources from virtual to host cluster, vCluster also copies the CustomResourceDefinitions from the host cluster to the virtual cluster to ensure compatibility.

vCluster does not require you to specify an API version when syncing CustomResourceDefinitions. When multiple API versions exist and you do not define a specific version, vCluster automatically uses the storage version. If your CustomResourceDefinition has multiple versions and you need a specific API version, specify it in your `vcluster.yaml` configuration.

```yaml
sync:
  toHost:
    customResources:
      certificates.cert-manager.io:
        enabled: true
```

## Versioned example

To use a specific API version instead of the default storage version, add the version after the "/" separator in your configuration. For example, to specify version `v1` of the `certificates.cert-manager.io` CustomResourceDefinition, use `certificates.cert-manager.io/v1`.

This is helpful when you need a specific version that is not the storage version, or when the storage version is unknown.

```yaml
sync:
  toHost:
    customResources:
      certificates.cert-manager.io/v1:
        enabled: true
```

:::info Unique CRD versions
vCluster can only sync one version of a CustomResource at a time. If you specify multiple versions, vCluster fails to start.
:::


## Patches

You can modify the sync behavior with patches that target specific paths. vCluster supports two different types of patches.

:::info Wildcard patches
You can use `*` in paths to select all entries of an array or object, such as `spec.containers[*].name` or `spec.containers[*].volumeMounts[*]`. vCluster calls the patch multiple times for each matching entry.
:::

### Reference patches

Reference patches make specific fields of one resource point to a different resource that gets rewritten. vCluster automatically imports the referenced resource to the virtual cluster when it finds the resource in the host cluster. For example:

```yaml title="Reference patch configuration"
sync:
  toHost:
    customResources:
      certificates.cert-manager.io:
        enabled: true
        patches:
        - path: spec.secretName
          reference:
            apiVersion: v1
            kind: Secret
```

vCluster translates the path `spec.secretName` because it points to a Secret. When the Secret is created in the host cluster, vCluster automatically imports it into the virtual cluster.

:::info Multi-Namespace-Mode
With multi-namespace-mode you only need to rewrite references that include a namespace. You can use the `namespacePath` option to specify the path of the namespace of the reference.
:::

### JavaScript expression patches {#javascript-expression-patches}
<!-- vale on -->

JavaScript expression patches use ES6 compatible expressions to change fields during sync. You define how fields change when syncing from the virtual cluster to the host cluster or when syncing from the host cluster to the virtual cluster. To add a suffix to certificate DNS names:

```yaml title="JavaScript expression patch configuration"
sync:
  toHost:
    customResources:
      certificates.cert-manager.io:
        enabled: true
        patches:
        - path: spec.dnsNames[*]
          # specifies the sync direction here again, because you can also react on change for fromHost with an expression
          expression: '"www."+value'
          # optional reverseExpression, if omitted patches from host are discarded for that path
          # reverseExpression: 'value.slice("my-prefix".length)'
```

Besides `value`, you can access a variable called `context` that contains vCluster specific data:
- `context.vcluster.name`: Name of the virtual cluster
- `context.vcluster.namespace`: Namespace of the virtual cluster
- `context.vcluster.config`: Config of the virtual cluster, basically `vcluster.yaml` merged with the defaults
- `context.hostObject`: Host object (can be null if not available)
- `context.virtualObject`: Virtual object (can be null if not available)
- `context.path`: The matched path on the object, useful when you use wildcard path selectors (*)

For example, to add `www.` to every DNS name specified in a cert-manager certificate in the path `spec.dnsNames`:

```yaml title="Complete DNS name patch example"
sync:
  toHost:
    customResources:
      certificates.cert-manager.io:
        enabled: true
        patches:
        - path: spec.dnsNames[*]
          # specifies the sync direction here again, because you can also react on change for fromHost
          expression: "value.startsWith('www.') ? value : `www.${value}`"
          # specifies how to sync back changes to the virtual cluster. If omitted does not sync back changes.
          reverseExpression: "value.startsWith('www.') ? value.slice(4) : value"
```

#### Example

When you create a new certificate within the vCluster:

```yaml title="Original certificate in virtual cluster"
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: name-within-vcluster
spec:
  dnsNames:
    - example.com
```

vCluster syncs to the host cluster and applies your patch, which creates this modified certificate:

```yaml title="Patched certificate in host cluster"
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: synced-name
spec:
  dnsNames:
    - www.example.com # the patch added www. to this field
```

When you directly edit the certificate in the host cluster and change the domain:

```yaml title="Modified certificate in host cluster"
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: synced-name
spec:
  dnsNames:
    - www.other-domain.com # changed from www.example.com
```

vCluster detects the change, applies the reverse patch, and updates the certificate in your virtual cluster:

```yaml title="Updated certificate in virtual cluster"
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: name-within-vcluster
spec:
  dnsNames:
    - other-domain.com # the patch removed the www. from www.other-domain.com
```

## Configure Kubernetes Gateway API sync

To use the Kubernetes Gateway API with custom resources, follow these steps:

### Install Gateway CRD in the host

```bash title="Install Gateway CRD"
kubectl --context="${HOST_CTX}" get crd gateways.gateway.networking.k8s.io &> /dev/null || \
kubectl --context="${HOST_CTX}" apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/v1.2.1/standard-install.yaml
```

### Create waypoint gateway {#create-waypoint-gateway}

Create a waypoint gateway configuration:

```yaml title="waypoint-gateway.yaml"
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: waypoint
  labels:
    istio.io/waypoint-for: service
spec:
  gatewayClassName: istio-waypoint
  listeners:
  - name: mesh
    port: 15008
    protocol: HBONE
```

Apply it to your host cluster:

```bash title="Create Waypoint Gateway"
kubectl --context="${HOST_CTX}" create -f waypoint-gateway.yaml --namespace="${VCLUSTER_HOST_NAMESPACE}"
```

Once configured, you can configure your custom resources to sync Gateway API resources between the virtual and host clusters.
