Each virtual cluster runs its own DNS service using CoreDNS, which works the same as standard Kubernetes. You can deploy CoreDNS in two ways: as a separate pod (default) or embedded within the syncer (enterprise feature).

Pods and services within each virtual cluster discover each other using DNS names such as `my-service.default.svc.cluster.local`. Each virtual cluster maintains its own DNS namespace, so applications in different virtual clusters cannot discover or access services outside their virtual cluster.

When a pod queries a service such as `my-app.default.svc.cluster.local`, CoreDNS returns the service IP address for normal service discovery.

## How DNS resolution works in virtual clusters

DNS resolution in virtual clusters works the same as standard Kubernetes clusters. Applications make DNS queries to resolve service names, and CoreDNS responds with the correct IP addresses for services within the virtual cluster.

When an application pod initiates a DNS query, the request travels to the virtual cluster's CoreDNS instance on port 1053. CoreDNS maintains a complete view of all services that exist within the virtual cluster and resolves queries to the appropriate service IP addresses.

## Deployment architectures

Virtual clusters support two CoreDNS deployment options.

### Deploy separate CoreDNS

A standard virtual cluster deployment consists of two pods per virtual cluster instance: the virtual cluster pod containing the API server and syncer components, and a separate CoreDNS pod. Each virtual cluster operates its own DNS service so that pods and services can locate each other by hostnames, with the syncer ensuring that standard Kubernetes DNS naming logic applies.

The separate deployment model isolates DNS operations from the API server and sync functions. CoreDNS listens on port 1053 instead of port 53 to avoid conflicts with the host cluster's DNS service. DNS issues can be debugged, updated, or restarted independently without affecting the virtual cluster's control plane availability. However, virtual cluster pods cannot directly access host cluster services via DNS due to isolation boundaries.

### Deploy embedded CoreDNS

<ProAdmonition/>

The embedded CoreDNS feature allows you to run CoreDNS as part of the syncer, which saves the overhead of an external CoreDNS pod. This enterprise feature consolidates all virtual cluster components into a single pod, reducing pod count and optimizing resource allocation.

The embedded approach requires consideration of operational trade-offs. Troubleshooting might become more complex since DNS and sync functions share the same container and process space. Updates and restarts affect both DNS and sync capabilities simultaneously, though all CoreDNS features remain supported.

## CoreDNS uses port 1053

CoreDNS in virtual clusters listens on port 1053 instead of the default DNS port 53 to avoid conflicts with the host cluster's DNS.

The host cluster already operates its own DNS service on the standard port 53, so the virtual cluster DNS services cannot bind to the same port. Since multiple virtual clusters might run on the same host cluster, each requires its own DNS service that operates independently.

Port 53 is classified as a privileged port on most operating systems and requires elevated permissions to bind to it. By using port 1053, CoreDNS containers can operate with standard user privileges.

### Consider cloud provider networking

Different cloud providers have varying network policies that can affect DNS traffic on port 1053.

EKS clusters might require security group configuration to allow inbound traffic on port 1053 between worker nodes.

GKE and AKS typically allow internal cluster traffic by default, but custom network policies may need to explicitly permit UDP traffic on port 1053 between cluster nodes.

## Recognize DNS isolation and limitations

Virtual cluster DNS isolation maintains clear boundaries between virtual clusters while creating specific challenges for cross-cluster communication.

### DNS boundaries

Virtual clusters cannot directly access host services via DNS, and host cluster pods cannot access virtual cluster services through the virtual cluster's DNS.

When a pod inside a virtual cluster attempts to resolve a DNS name for a service that exists only in the host cluster, the virtual cluster CoreDNS instance returns an NXDOMAIN response because it maintains no knowledge of host cluster resources.

### Apply service mapping strategies

External name services provide an approach for mapping host cluster services into a virtual cluster's DNS namespace. This pattern creates a virtual cluster service that acts as a DNS alias.

```yaml title="External name service configuration"
apiVersion: v1
kind: Service
metadata:
  name: database-proxy
  namespace: my-application
spec:
  type: ExternalName
  externalName: postgres.data-tier.svc.cluster.local
  ports:
  - port: 5432
    targetPort: 5432
    protocol: TCP
```

This configuration creates a service named `database-proxy` that applications can access using normal Kubernetes DNS patterns. When applications connect to `database-proxy.my-application.svc.cluster.local`, the DNS resolution returns the host cluster service `postgres.data-tier.svc.cluster.local`.

Resource synchronization imports actual host cluster services into the virtual cluster namespace.

```yaml title="Resource synchronization configuration"
sync:
  fromHost:
    services:
      enabled: true
      selector:
        matchLabels:
          vcluster.loft.sh/import: "true"
    endpoints: true
```

## Configure CoreDNS

CoreDNS configuration depends on your deployment requirements. The following configurations cover common deployment scenarios.

### Set up basic CoreDNS

Basic configuration sets resource limits and replica count:

```yaml title="Basic CoreDNS configuration"
controlPlane:
  coredns:
    enabled: true
    replicas: 1
    resources:
      requests:
        cpu: 20m
        memory: 32Mi
      limits:
        cpu: 100m
        memory: 128Mi
```

### Configure private registry

For deployments requiring private registry images:

```yaml title="Private registry CoreDNS configuration"
controlPlane:
  coredns:
    image:
      repository: registry.company.com/security/coredns
      tag: "1.10.1-hardened"
      pullPolicy: Always
    imagePullSecrets:
      - name: company-registry-credentials
```

### Configure custom CoreDNS

Override default CoreDNS configuration to modify DNS behavior:

```yaml title="Custom CoreDNS configuration"
controlPlane:
  coredns:
    overwriteConfig: |
      .:1053 {
          errors
          health
          ready
          kubernetes cluster.local in-addr.arpa ip6.arpa {
              pods insecure
              fallthrough in-addr.arpa ip6.arpa
              ttl 30
          }
          prometheus :9153
          forward . /etc/resolv.conf
          cache 30
          loop
          reload
      }
```

### Configure embedded CoreDNS

The embedded CoreDNS feature runs CoreDNS within the syncer container:

```yaml title="Embedded CoreDNS enterprise configuration"
controlPlane:
  coredns:
    enabled: true
    embedded: true
    resources:
      requests:
        cpu: 10m
        memory: 16Mi
      limits:
        cpu: 50m
        memory: 64Mi
```
