# Configure CORS policy for the API server
<!-- vale Google.Headings = YES -->

Control which browser-based origins can make cross-origin requests to the <GlossaryTerm term="vcluster">vCluster</GlossaryTerm> <GlossaryTerm term="api-server">API server</GlossaryTerm> by configuring Cross-Origin Resource Sharing (CORS) allowed origins.

## Prerequisites

<BasePrerequisites />

- An ingress controller (such as ingress-nginx) deployed on the host cluster.

## Overview

Browsers enforce the [same-origin policy](https://developer.mozilla.org/en-US/docs/Web/Security/Same-origin_policy) for cross-origin requests. When browser-based tools such as dashboards or monitoring UIs connect directly to the virtual cluster API server, requests from a different origin are blocked unless the server includes the correct CORS response headers.

The platform doesn't set CORS headers by default. Since the vCluster Platform is self-hosted, you must configure CORS manually if you expose the API server through an [ingress](../../../../configure/vcluster-yaml/control-plane/deployment/ingress.mdx) or [AccessPoint](../../../../key-features/ingress-access.mdx) and browser-based clients need access. Configure your ingress controller to add CORS headers at the network layer.

:::info When CORS configuration is needed
Most `kubectl` and in-cluster service-to-service traffic doesn't involve browsers, so CORS headers are irrelevant. Configure CORS only when browser-based clients connect directly to the API server from a different origin.
:::

## Configure ingress-level CORS {#ingress-cors}

Configure your ingress controller to add CORS headers. This requires TLS termination at the ingress rather than the default SSL passthrough.

### ingress-nginx

Override the default vCluster ingress annotations to disable SSL passthrough and enable CORS:

```yaml title="vcluster.yaml"
controlPlane:
  ingress:
    enabled: true
    host: "vcluster.example.com"
    annotations:
      # TLS termination instead of passthrough
      nginx.ingress.kubernetes.io/backend-protocol: "HTTPS"
      nginx.ingress.kubernetes.io/ssl-redirect: "true"
      # CORS
      nginx.ingress.kubernetes.io/enable-cors: "true"
      nginx.ingress.kubernetes.io/cors-allow-origin: "https://dashboard.example.com"
      nginx.ingress.kubernetes.io/cors-allow-methods: "GET, POST, PUT, DELETE, PATCH, OPTIONS"
      nginx.ingress.kubernetes.io/cors-allow-headers: "Authorization, Content-Type"
      nginx.ingress.kubernetes.io/cors-allow-credentials: "true"
      nginx.ingress.kubernetes.io/cors-max-age: "3600"
    spec:
      tls:
        - hosts:
            - vcluster.example.com
          secretName: vcluster-tls
```

:::warning SSL passthrough must be disabled
The default vCluster ingress uses `ssl-passthrough: true`, which forwards encrypted traffic directly to the API server. With SSL passthrough, the ingress controller can't read or modify HTTP headers, so CORS annotations are ignored. The configuration above omits `ssl-passthrough` and provides a TLS secret for the ingress to end TLS itself.
:::

### Annotation reference

All annotations use the `nginx.ingress.kubernetes.io/` prefix.

| Annotation | Description | Default |
|---|---|---|
| `enable-cors` | Enable CORS headers | `"false"` |
| `cors-allow-origin` | Allowed origins (comma-separated) | `"*"` |
| `cors-allow-methods` | Allowed HTTP methods | `"GET, PUT, POST, DELETE, PATCH, OPTIONS"` |
| `cors-allow-headers` | Allowed request headers | Standard browser headers and `Authorization` |
| `cors-allow-credentials` | Allow credentials (cookies, auth) | `"false"` |
| `cors-expose-headers` | Response headers exposed to browser | (none) |
| `cors-max-age` | Preflight cache duration in seconds | `"1728000"` |

:::warning
Don't set `cors-allow-origin` to `"*"` when `cors-allow-credentials` is `"true"`. Browsers reject this combination per the CORS specification. List explicit origins instead.
:::

### Other ingress controllers

Traefik and Istio support CORS through their own configuration mechanisms:

- **Traefik**: Use the [headers middleware](https://doc.traefik.io/traefik/middlewares/http/headers/) with `accessControlAllowOriginList` on an IngressRoute.
- **Istio**: Use the [`corsPolicy` field](https://istio.io/latest/docs/reference/config/networking/virtual-service/) on a VirtualService.
- **Gateway API**: The experimental CORS filter (Gateway API v1.3+) supports CORS on HTTPRoute resources. See the [Gateway API CORS guide](https://gateway-api.sigs.k8s.io/guides/http-cors/) for details.

## Security considerations

### Avoid overly broad origins

Setting `cors-allow-origin` to `"*"` allows every origin and disables CORS protection entirely. This exposes the API server to cross-site request attacks from any website a user visits.

:::danger
Never use `cors-allow-origin: "*"` in production. Restrict origins to the specific domains that need access.
:::

### Principle of least privilege

- List only the exact origins that need browser-based API access.
- Prefer specific origins (`https://dashboard.example.com`) over wildcards.
- Separate development and production configurations. Localhost origins shouldn't appear in production values.

<!-- vale Google.Headings = NO -->
### CORS doesn't replace authentication
<!-- vale Google.Headings = YES -->

CORS headers control which origins a browser permits for cross-origin requests. They don't authenticate or authorize API calls. The virtual cluster's RBAC, authentication tokens, and network policies remain the primary access controls.

<!-- vale Google.Headings = NO -->
## Troubleshoot CORS issues
<!-- vale Google.Headings = YES -->

### Browser shows "blocked by CORS policy"

The ingress didn't return an `Access-Control-Allow-Origin` header for the requesting origin. Verify the following:

1. The `enable-cors` annotation is set to `"true"` on the ingress.
2. The `cors-allow-origin` value matches the exact origin the browser sends (scheme, host, and port must all match).
3. SSL passthrough is disabled so the ingress controller can modify response headers.

### Test with curl

Send a preflight request to confirm the ingress returns CORS headers:

```bash
curl -v -X OPTIONS \
  -H "Origin: https://dashboard.example.com" \
  -H "Access-Control-Request-Method: GET" \
  https://<api-server-url>/api/v1/namespaces
```

A correct response includes these headers:

```
Access-Control-Allow-Origin: https://dashboard.example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, PATCH, OPTIONS
Access-Control-Allow-Headers: Authorization, Content-Type
```

If these headers are missing, check the ingress annotations and confirm SSL passthrough is disabled.

## Limitations

- Ingress-level CORS requires TLS termination at the ingress controller, which changes the TLS certificate presented to clients from the API server certificate to the ingress certificate.
- Changes to ingress annotations require an upgrade of the virtual cluster (`vcluster create --upgrade` or `helm upgrade`) for the ingress resource to be updated.
