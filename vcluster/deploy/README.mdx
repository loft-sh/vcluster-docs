Every vCluster deployment is shaped by two independent decisions: where the control plane runs, and how worker nodes are deployed. Both choices must be made before you deploy — neither can be changed on an existing cluster without redeploying.

## Decision 1: Control plane

The control plane is the management layer for each tenant cluster. It runs in one of three modes depending on your infrastructure:

| Mode | Infrastructure | Best for |
|------|---------------|----------|
| [Kubernetes pod](./control-plane/kubernetes-pod/basics.mdx) | Existing Kubernetes cluster | Most deployments; you already operate a cluster |
| [Standalone (binary)](./control-plane/binary/README.mdx) | Bare metal or VMs; no Kubernetes required | AI cloud providers; bare-metal platforms; "cluster one" bootstrapping |
| [Docker container (vind)](./control-plane/docker-container/basics.mdx) | Docker only; no Kubernetes required | Local development; CI/CD pipelines |

## Decision 2: Worker nodes

Worker nodes host the actual tenant workloads. You choose between tenant clusters that share existing compute and tenant clusters that run on dedicated nodes:

| Type | Node source | Best for |
|------|-------------|----------|
| [Shared nodes](./worker-nodes/host-nodes/README.mdx) | Nodes from the Control Plane Cluster's existing pool | Internal platforms; CI/CD; development environments |
| [Private nodes](./worker-nodes/private-nodes/README.mdx) | Nodes that join each tenant cluster exclusively | GPU workloads; regulated environments; full infrastructure isolation |

:::warning
Shared and private nodes cannot be mixed within a single vCluster. Private nodes must be enabled at creation time. An existing vCluster using shared nodes cannot be upgraded to use private nodes — you must deploy a new cluster.
:::

## Deployment paths

The two decisions combine into four deployment paths:

| Control plane | Worker nodes | Typical use case |
|--------------|--------------|------------------|
| Kubernetes pod | Shared nodes | High-density tenant isolation; developer platforms; CI |
| Kubernetes pod | Private nodes | Regulated tenants; GPU pools; enterprise isolation |
| Standalone (binary) | Private nodes | AI cloud operators; bare-metal platforms; provisioning at hyperscaler speed |
| Docker container (vind) | Private nodes (Docker) | Local development; offline CI; hybrid cloud experiments |

vind always uses private nodes — worker nodes are Docker containers provisioned automatically.

## Configuration locked at deploy time

Some settings cannot be changed by upgrading an existing vCluster. Decide these before you deploy:

- **Worker node type** — shared nodes or private nodes
- **Backing store** — embedded SQLite (default), etcd, MySQL, or PostgreSQL
- **Control plane distro** — the Kubernetes distribution used by the control plane

Everything else can be updated by upgrading the vCluster with a new `vcluster.yaml`.
