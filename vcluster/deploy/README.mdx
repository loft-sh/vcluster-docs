## I have a Kubernetes cluster

The control plane runs as a pod on any conformant Kubernetes cluster, including EKS, GKE, AKS, OpenShift, and self-managed clusters.

<figure>
  <DeployKubernetes style={{width: '100%', maxWidth: '600px', height: 'auto', margin: '0 auto', display: 'block'}} role="img" aria-label="Tenant clusters running as pods inside a host Kubernetes cluster." />
</figure>

You choose how to provision worker nodes for each tenant cluster:

- **[Shared nodes](./worker-nodes/host-nodes/README.mdx)** — tenant workloads run on the Control Plane Cluster's existing node pool. Good for internal platforms, CI/CD, and development environments.
- **[Private nodes](./worker-nodes/private-nodes/README.mdx)** — dedicated nodes join each tenant cluster exclusively. Ideal for GPU workloads, regulated environments, and full isolation.

You can't mix shared and private nodes in a single vCluster, and you can't change the node type after deployment.

:::warning
**Decide before you deploy.** vCluster locks the worker node type, backing store (SQLite, etcd, MySQL, or PostgreSQL), and control plane distro at creation time. You can't change them by upgrading an existing vCluster.
:::

[Deploy on a Kubernetes cluster →](./control-plane/kubernetes-pod/basics.mdx)

## I have bare metal or VMs

[vCluster Standalone](./control-plane/binary/README.mdx) deploys a full Kubernetes cluster directly on your nodes. No existing cluster required, minimal setup, no heavyweight distribution overhead. Add nodes as needed and have a working cluster in minutes.

<figure>
  <DeployStandalone style={{width: '100%', maxWidth: '600px', height: 'auto', margin: '0 auto', display: 'block'}} role="img" aria-label="A Standalone cluster running as a binary directly on bare metal or VMs with no host cluster layer." />
</figure>

Once the cluster is up, join worker nodes from any infrastructure. Nodes connect to the exposed cluster endpoint using a bootstrap token, so workers don't need to be co-located. Bring in cloud provider instances, VMs from a different site, or additional bare metal machines. See [Private nodes](./worker-nodes/private-nodes/README.mdx) for requirements and steps. To add vCluster tenant clusters on top, follow the [Kubernetes path](#i-have-a-kubernetes-cluster) above.

:::info
**Decide before you deploy.** vCluster sets the backing store at install time. You can't change it without redeploying.
:::

[Deploy Standalone →](./control-plane/binary/README.mdx)

## I am using Docker

[vind](./control-plane/docker-container/basics.mdx) runs a full Kubernetes cluster entirely in Docker. The control plane and worker nodes all run as containers. Docker is the only requirement.

<figure>
  <DeployDocker style={{width: '100%', maxWidth: '600px', height: 'auto', margin: '0 auto', display: 'block'}} role="img" aria-label="A vind cluster running control plane and worker nodes as Docker containers on a laptop or CI runner." />
</figure>

Use vind for local development, offline CI pipelines, and testing configurations before committing to a cluster deployment.

:::warning
**Decide before you deploy.** vCluster sets the backing store and control plane distro at creation time. You can't change them without redeploying.
:::

[Deploy with Docker →](./control-plane/docker-container/basics.mdx)
