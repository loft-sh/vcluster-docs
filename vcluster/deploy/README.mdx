Two decisions shape every vCluster deployment: where the control plane runs and how you deploy worker nodes. Make both choices before you deploy. You cannot change either on an existing cluster without redeploying.

## Decision 1: Control plane

The control plane is the management layer for each tenant cluster. It runs in one of three modes depending on your infrastructure.

| Mode | Infrastructure | Best for |
|------|---------------|----------|
| [Kubernetes pod](./control-plane/kubernetes-pod/basics.mdx) | Existing Kubernetes cluster | Most deployments where you already operate a cluster |
| [Standalone (binary)](./control-plane/binary/README.mdx) | Bare metal or VMs, no Kubernetes required | AI cloud providers, bare-metal platforms, bootstrapping without an existing cluster |
| [Docker container (vind)](./control-plane/docker-container/basics.mdx) | Docker only, no Kubernetes required | Local development, CI/CD pipelines |

## Decision 2: Worker nodes

Worker nodes host tenant workloads. Choose between tenant clusters that share existing compute and tenant clusters that run on dedicated nodes.

| Type | Node source | Best for |
|------|-------------|----------|
| [Shared nodes](./worker-nodes/host-nodes/README.mdx) | Nodes from the Control Plane Cluster's existing pool | Internal platforms, CI/CD, development environments |
| [Private nodes](./worker-nodes/private-nodes/README.mdx) | Nodes that join each tenant cluster exclusively | GPU workloads, regulated environments, full infrastructure isolation |

:::warning
You cannot mix shared and private nodes in a single vCluster. Enable private nodes at creation time. You cannot upgrade an existing vCluster from shared to private nodes. Deploy a new cluster instead.
:::

## Deployment paths

Your choices combine into four deployment paths.

| Control plane | Worker nodes | Typical use case |
|--------------|--------------|------------------|
| Kubernetes pod | Shared nodes | High-density tenant isolation, developer platforms, CI |
| Kubernetes pod | Private nodes | Regulated tenants, GPU pools, enterprise isolation |
| Standalone (binary) | Private nodes | AI cloud operators, bare-metal platforms, hyperscaler-speed provisioning |
| Docker container (vind) | Private nodes (Docker) | Local development, offline CI, hybrid cloud experiments |

vind always uses private nodes. It provisions worker nodes as Docker containers automatically.

## Configuration locked at deploy time

You cannot change these settings by upgrading an existing vCluster. Decide before you deploy.

- **Worker node type**: shared nodes or private nodes
- **Backing store**: embedded SQLite (default), etcd, MySQL, or PostgreSQL
- **Control plane distro**: the Kubernetes distribution used by the control plane

Upgrade the vCluster with a new `vcluster.yaml` to update everything else.
