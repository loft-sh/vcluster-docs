No Kubernetes cluster? No problem. vind (vCluster in Docker) runs a complete Kubernetes cluster entirely in Docker containers — no existing infrastructure required.

Beyond what KinD offers, vind adds capabilities that matter for real development workflows:

| Feature | vind | KinD |
|---|---|---|
| Pause and resume without losing state | Yes | No — must delete and recreate |
| LoadBalancer services | Automatic | Manual setup required |
| Virtual worker nodes | Yes | Limited |
| Join real cloud instances | Yes, via VPN | No |
| Pull-through image cache | Uses host Docker daemon | Direct registry pulls |

## Prerequisites

- Docker installed and running
- At least 2 GB RAM available to Docker

## Deploy a cluster

:::note Load balancer services on macOS
Automatic LoadBalancer support requires elevated privileges on macOS with Docker Desktop or Rancher Desktop. Run with `sudo` to enable it:

```bash
sudo vcluster create my-vcluster
```
:::

## Pause and resume

A vind cluster can pause and resume without losing state. A KinD cluster requires a full delete and recreate.

```bash
# Pause the cluster — frees Docker resources immediately
vcluster pause my-vcluster

# Check the status
vcluster list
```

Output is similar to:

```
NAME          NAMESPACE   STATUS   AGE
my-vcluster               Paused   5m
```

```bash
# Resume — cluster is back in seconds
vcluster resume my-vcluster
vcluster connect my-vcluster
```

Paused clusters retain all their state. After resume, workloads restart where they left off.

## LoadBalancer services
<!-- vale Google.Headings on -->

LoadBalancer services work automatically, with no MetalLB or extra configuration.

```bash
kubectl create deployment hello --image=nginx
kubectl expose deployment hello --port=80 --type=LoadBalancer
kubectl get service hello --watch
```

Output is similar to:

```
NAME    TYPE           CLUSTER-IP    EXTERNAL-IP   PORT(S)        AGE
hello   LoadBalancer   10.96.x.x     127.0.0.1     80:31234/TCP   5s
```

```bash
curl http://localhost
```

The service is reachable from your host immediately. On Linux, this works without elevated privileges. On macOS, run `vcluster create` with `sudo` at cluster creation time (see above).

## Add virtual worker nodes

Test node affinity rules, DaemonSet scheduling, or topology spread constraints without real hardware. Add virtual worker nodes by configuring them in a `vcluster.yaml` at creation time.

```yaml title="vcluster.yaml"
experimental:
  docker:
    nodes:
      - name: worker-1
      - name: worker-2
```

```bash
vcluster create my-cluster --values vcluster.yaml
kubectl get nodes
```

Output is similar to:

```
NAME        STATUS   ROLES           AGE
my-cluster  Ready    control-plane   30s
worker-1    Ready    <none>          25s
worker-2    Ready    <none>          25s
```

Schedule workloads to specific nodes, add labels and taints, or simulate a realistic multi-node topology — all from a single Docker host.

## Join real cloud nodes

vind clusters accept real cloud instances as worker nodes through vCluster VPN. An EC2 instance, GCP VM, or any Linux machine can join your local vind cluster through a VPN tunnel, giving you a hybrid cluster: local control plane, real remote compute.

This requires vCluster Platform. See [vCluster VPN documentation](../configure/vcluster-yaml/private-nodes/vpn.mdx) for setup instructions.

## Clean up

```bash
vcluster delete my-vcluster
```

All Docker containers and networks created for the cluster are removed.
