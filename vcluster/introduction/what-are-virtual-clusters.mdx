vCluster is a production-grade Kubernetes platform that lets AI clouds and enterprises provision and manage tenant clusters at hyperscaler scale. Each tenant receives a fully isolated Kubernetes environment backed by a virtualized control plane—with no shared API attack surface, no in-cluster agent pods, and complete separation from other tenants.

vCluster supports the full spectrum of isolation requirements, from cost-efficient shared-node configurations to dedicated hardware with private GPUs and bare metal. Environments are defined declaratively and provision in seconds, giving operators the automation and reliability of a hyperscaler without the overhead of managing separate physical clusters per tenant.

## Tenant clusters and the virtualized control plane

A tenant cluster is a fully functional Kubernetes environment with its own API server, controller manager, and resource namespace. From a tenant’s perspective it behaves exactly like a standard Kubernetes cluster—supporting custom resources, RBAC, workload scheduling, and any conformant tooling.

Behind the scenes, each tenant cluster’s control plane is virtualized and hosted on a **Control Plane Cluster** operated by the provider. The Control Plane Cluster is completely invisible to tenants: there are no shared control plane nodes, no in-cluster agent pods, and no lateral path between tenant environments.

vCluster supports the full spectrum of tenancy configurations—from shared-node setups optimized for density to private-node configurations with dedicated GPUs and bare metal for workloads requiring hardware-level isolation.

Tenant clusters are certified Kubernetes distributions, adhering to upstream standards while maintaining strict isolation from the Control Plane Cluster and neighboring tenants.

This flexibility lets you select the right isolation model for your security, cost, and performance requirements, while benefiting from rapid provisioning and centralized lifecycle management.


## Tenancy models overview

vCluster supports a range of tenancy models, allowing you to choose the right balance of isolation, cost-efficiency, and operational complexity for your platform.
Each tenant cluster is backed by a Control Plane Cluster, but how it isolates workloads, consumes resources, and interacts with the underlying infrastructure depends on the tenancy model you select.

Below are the tenancy models supported by vCluster:

### Shared Nodes
* The tenant cluster's control plane runs as a containerized process on the Control Plane Cluster.
* Worker nodes are drawn from the Control Plane Cluster's shared node pool.

#### How it works

All tenant clusters run against a single Control Plane Cluster and schedule pods onto the same shared node pool. The virtualized control plane enforces separation at the API, RBAC, and CRD levels, but does not restrict pod scheduling unless additional mechanisms (e.g., taints, affinities) are applied.
Tenants interact with their own cluster as if it is a fully separate environment, but their workloads run side-by-side with those from other tenants at the node level. Shared infrastructure components like the container runtime, CNI, and CSI drivers are used across all tenants.

### Dedicated Nodes
* The tenant cluster's control plane runs as a containerized process on the Control Plane Cluster.
* Worker nodes are drawn from a specific labeled subset of the Control Plane Cluster's node pool.

#### How it works

Each tenant cluster is configured with a Kubernetes nodeSelector (or affinity rules) that ensures all tenant workloads are scheduled only to nodes with specific labels. For example, a tenant cluster assigned to `nodegroup=team-a` will only run pods on nodes matching that label.

While compute is scoped to these dedicated nodes, all other components—like the CNI, CSI, and the Control Plane Cluster itself—remain shared. The virtualized control plane maintains full API isolation, separate CRDs, tenant-specific RBAC, and control plane security.

### Virtual Nodes
* The tenant cluster's control plane runs as a containerized process on the Control Plane Cluster.
* Worker nodes of the tenant cluster are running [vNode](https://www.vnode.com/docs/) on nodes within the Control Plane Cluster.

#### How it works
Virtual Nodes are implemented within vCluster by inserting a translation layer between the virtualized control plane and the physical cluster. From the perspective of a tenant, the tenant cluster presents one or more virtual nodes, each representing a safe, scoped execution environment.

Internally, workloads from these virtual nodes are scheduled as regular pods on shared physical nodes, but the tenant does not see or interact with the real underlying node environment. This abstraction enables stronger tenant isolation, enforces placement boundaries, and prevents tenants from seeing or interacting with each other’s workloads—even though compute is technically shared.

### Private Nodes
* The tenant cluster's control plane runs as a containerized process on a dedicated Control Plane Cluster.
* Worker nodes are individual nodes not connected to any other Kubernetes cluster. You can either join manually provisioned nodes to a tenant cluster or configure vCluster to automatically provision and join nodes based on resource requirements.

#### How it works

Each tenant cluster is backed by a dedicated Control Plane Cluster provisioned with a private set of physical nodes. The CNI, CSI, kube-proxy, and all other Kubernetes components are fully isolated per tenant.

This model combines the benefits of a virtualized control plane (rapid provisioning, CRD freedom, sleep mode, etc.) with a hard isolation boundary at the infrastructure level. Tenants cannot interfere with one another’s environments at any layer—from API server to node kernel.

### vCluster Standalone
<!-- vale on -->
* The tenant cluster's control plane is deployed as a binary on individual nodes.
* Worker nodes are individual nodes that are not part of any other Kubernetes cluster.

#### How it works

vCluster Standalone launches its own lightweight control plane components (like an API server and scheduler) and backs workloads with a local Kubernetes-compatible runtime. No Control Plane Cluster is required—tenant clusters can run independently in a container, virtual machine, or even on a developer laptop.
Despite being self-contained, vCluster Standalone behaves just like a regular Kubernetes control plane: it supports custom resources, RBAC, workload scheduling, and integrations with CI/CD tools. Workloads are managed entirely within the tenant cluster.
