Running Kubernetes for multiple teams, customers, or GPU workloads means managing tenant isolation at scale. Dedicated clusters per tenant are expensive and operationally heavy. Shared clusters create noisy neighbors and hard-to-enforce boundaries. vCluster solves this without compromise.

vCluster provisions fully isolated Kubernetes environments, called <GlossaryTerm term="tenant-cluster">tenant clusters</GlossaryTerm>, on top of existing infrastructure. Each tenant gets a Kubernetes API server, their own CRDs, RBAC, and resource namespace. From the tenant's perspective, it behaves exactly like a dedicated cluster. For the operator, it provisions in seconds and is centrally managed.

## How it works

Each tenant cluster runs its own virtualized control plane as a containerized process, with a dedicated API server, controller manager, and data store. That control plane is hosted on a <GlossaryTerm term="control-plane-cluster">Control Plane Cluster</GlossaryTerm> that you operate. The Control Plane Cluster is completely invisible to tenants. There are no shared control plane nodes, no in-cluster agent pods, and no lateral path between tenant environments.

When a tenant deploys a workload, the control plane accepts the request and a <GlossaryTerm term="syncer">syncer</GlossaryTerm> component translates it to the underlying infrastructure. The tenant never sees the Control Plane Cluster. They only see their cluster.

Tenant clusters are certified Kubernetes distributions. Any conformant tool works against them without modification: `kubectl`, Helm, Argo, Crossplane, and others.

## Tenancy models

The tenancy model determines how isolated each tenant's environment is and how compute is consumed. vCluster supports the full isolation spectrum:

| Model | Isolation level | Best for |
|---|---|---|
| **Private Nodes** | Full infrastructure isolation — dedicated nodes, CNI, CSI | GPU tenants, compliance workloads, regulated environments |
| **Dedicated Nodes** | Compute scoped to labeled node pools | Per-tenant GPU node pools without separate clusters |
| **Virtual Nodes** | Node-level isolation semantics over shared compute | Isolated GPU scheduling without dedicated hardware |
| **Shared Nodes** | API, RBAC, and CRD isolation; shared compute | High-density internal environments, CI/CD |

The tenancy model is set at creation time. See [Architecture](./architecture.mdx) for a detailed breakdown of each model.

:::note vCluster Standalone
[vCluster Standalone](./architecture.mdx#vcluster-standalone) runs without a Control Plane Cluster. It is suited for single-tenant deployments, edge, and air-gapped environments.
:::
